Every time someone opens a browser and visits a website, sends an email, or uses a mobile app, they’re relying on DNS.

This invisible system has processed thousands of requests on their behalf today alone, translating human-friendly names like google.com into the numerical IP addresses that computers actually use to communicate. Without DNS, the modern internet would be unusable.

DNS, or Domain Name System, is often compared to a phone book. Just as a phone book translates names into phone numbers, DNS translates domain names into IP addresses. This analogy captures the basic concept, but DNS is far more sophisticated. Unlike a static phone book, DNS is a massive, distributed, real-time database that handles billions of requests per second globally. It routes traffic across continents, enables load balancing across servers, and provides the foundation for how services find each other on the internet.

The critical nature of DNS became painfully clear in October 2016 when a cyberattack on Dyn, a major DNS provider, knocked out access to Twitter, Netflix, Reddit, and dozens of other major services. The websites themselves were functioning perfectly, but without DNS to translate their names into addresses, users couldn’t reach them. It was like having every street sign in a city suddenly disappear.

In this article, we will look at the journey of a DNS query in detail, the various DNS records, and the role of Anycast in DNS.

Journey of a DNS Query

When someone types “example[dot]com” into their browser and hits enter, a complex journey begins that typically completes in milliseconds. There is a hierarchy of server domains that need to be resolved.

Understanding this journey reveals how the internet actually finds and connects to websites. Here are the steps:

  • The browser’s first move is to check its own DNS cache. Chrome, Firefox, and other browsers maintain their own temporary storage of recently visited domains. If the same site was visited minutes ago, the browser already knows that “example[dot]com” translates to 93.184.216.34 and immediately initiates the connection.

  • If the browser’s cache doesn’t have the answer, it asks the operating system. Windows, macOS, and Linux all maintain their own DNS caches that are shared across all applications.

  • When neither the local cache nor the hosts file has the answer, the operating system contacts a recursive resolver. This external server acts as a detective that will track down the IP address, no matter how many other servers it needs to query. Most people use their ISP’s recursive resolver automatically. These servers typically sit in regional data centers within 10 to 50 miles of their users. Alternatively, many people configure their devices to use public resolvers like Google or Cloudflare. Corporate networks usually run their own resolvers in company data centers, providing additional control and security.

  • The resolver is called “recursive” because it handles the entire lookup process recursively, following referrals from server to server until it finds the answer. The resolver also maintains massive caches, storing responses for millions of domains. Popular sites like Google and Facebook are almost certainly in cache, serving instant responses to thousands of users without needing to perform the full lookup process.

  • When the recursive resolver doesn’t have a cached answer, it starts at the top of the DNS hierarchy with the root servers. Only 13 root server identities exist, labeled A through M, though each represents many physical servers distributed globally. These root servers have a simple but critical job. They don’t know the IP address for the example[dot]com, but they know who’s responsible for all COM domains. The root zone file they serve is surprisingly small, essentially just a list of TLD servers and their IP addresses.

  • The root server responds with something like “I don’t know about example[dot]com, but here are the servers that handle COM domains” and provides the IP addresses of the COM TLD servers. With this information, the recursive resolver moves to the next step.

  • The TLD servers maintain information about every registered domain under their TLD. They don’t store all the DNS records for every website, just enough to point to the authoritative name servers for each domain. When queried about example[dot]com, the TLD server responds with NS records indicating which servers are authoritative for that domain, perhaps “ns1[dot]exampledns[dot]com” and “ns2[dot]exampledns[dot]com.”

  • This creates an interesting problem. To contact ns1[dot]exampledns[dot]com, we need its IP address, but that might require another DNS lookup. The solution is glue records, where the TLD server includes the IP addresses of the name servers directly in its response, preventing circular dependencies.

  • Finally, the recursive resolver queries the authoritative name servers for example.com. These servers have the complete zone data for the domain and provide the definitive answer. They return the A record containing the IP address, and the recursive resolver begins the journey back.

  • The response travels back through the chain, getting cached at each level. The recursive resolver caches it based on the TTL (Time To Live) value. The operating system caches it in its resolver cache. The browser stores it in its cache.

  • This layered caching means the first user to look up a domain does the full journey, but subsequent users benefit from progressively faster responses as caches fill. A popular domain might be resolved entirely from the ISP’s recursive resolver cache, returning in under 10 milliseconds, while an obscure domain requires the full 50-100 millisecond journey through all the DNS servers.

DNS Records

DNS records are the individual pieces of information that DNS servers store and serve. Each record type serves a specific purpose, and understanding them is essential for managing domains and debugging issues.

Let’s look at them in more detail:

A Record

A records are the most fundamental DNS record type, mapping a domain name to an IPv4 address.

Despite IPv4 addresses running out years ago, A records remain critical because most of the internet still runs on IPv4. When example[dot]com has an A record pointing to a particular IP address, it tells the world that this domain lives at that IP address.

Websites often have multiple A records for the same domain, enabling basic load distribution.

AAAA Record

AAAA records (pronounced “quad-A”) serve the same purpose for IPv6 addresses.

IPv6 was designed to solve the address exhaustion problem with its vastly larger address space, but adoption has been slow. Many websites now include both A and AAAA records, allowing connections over either protocol. An AAAA record looks like “2606:2800:220:1:248:1893:25c8:1946”, which is considerably longer than IPv4 addresses.

CNAME Record

CNAME records create aliases, pointing one domain name to another.

The most common use is making www[dot]example[dot]com point to example[dot]com through a CNAME record. This means any changes to example[dot]com’s IP address automatically apply to www[dot]example[dot]com as well.

CNAMEs are particularly valuable for CDN integration. When a website uses Cloudflare, the subdomain might CNAME to something like “cdn[dot]cloudflare[dot]net,” letting Cloudflare control the actual IP addresses returned based on geographic location and server health.

NS Records

NS records establish the DNS delegation hierarchy. They indicate which servers are authoritative for a domain or subdomain.

At the TLD level, NS records for a domain like example[dot]com point to the authoritative name servers for that domain. Companies also use NS records to delegate control of subdomains to different providers. For example, a large organization might have example[dot]com on Cloudflare’s servers, delegate api[dot]example[dot]com to AWS Route 53 by creating NS records pointing to AWS’s name servers, and delegate mail[dot]example[dot]com to Google’s servers.

This separation allows different teams to manage their own DNS without interfering with each other. The glue records mentioned earlier solve the circular dependency when name servers are within the domain they serve.

MX Records

MX records handle email routing, specifying which servers receive email for a domain.

Each MX record includes a priority value, with lower numbers indicating higher priority. A typical setup might have “10 mail1[dot]example[dot]com” and “20 mail2[dot]example[dot]com,” meaning email goes to mail1 unless it’s unavailable, then falls back to mail2.

MX records ultimately rely on A records (or AAAA records for IPv6) to function properly. An MX record doesn’t directly contain an IP address. Instead, it points to a hostname, which then needs to be resolved to an IP address through an A or AAAA record.

TXT Records

TXT records store arbitrary text data and serve various purposes.

Domain verification is a common use, where services like Google Search Console ask website owners to add a specific TXT record to prove domain ownership. Email authentication heavily relies on TXT records for SPF (Sender Policy Framework) records that specify which servers can send email for a domain, DKIM records that store cryptographic signatures, and DMARC policies that tell receiving servers how to handle authentication failures.

TTL

TTL, while not a record type itself, is a critical value attached to every DNS record.

Specified in seconds, TTL tells recursive resolvers how long to cache the record. A TTL of 300 means five minutes, while 86400 means 24 hours. Short TTLs allow quick changes but increase query load on authoritative servers. Long TTLs improve performance, but changes take longer to propagate globally.

Anycast: The Magic Behind Modern DNS Scale

Traditional networking follows a simple rule: every IP address belongs to exactly one device in one location. For example, if a server in New York has the IP address 192.0.2.1, that address uniquely identifies that single machine.

Anycast breaks this assumption by allowing the same IP address to exist on hundreds of servers simultaneously around the world. This setup works due to the Internet’s routing protocols.

The secret lies in BGP (Border Gateway Protocol), which is how networks announce which IP addresses they can handle.

With anycast, operators place servers in multiple locations and configure them all with the same IP address. Each location then uses BGP to announce to nearby networks that it can handle traffic for that IP.

For example, networks in Asia learn about the Tokyo server, European networks learn about the London server, and American networks learn about the New York server, all claiming to serve the same IP address. When someone in Tokyo queries an IP address, their request travels through their ISP’s network, which has received BGP announcements from multiple Cloudflare locations. The internet’s routing system automatically chooses the “best” path, primarily based on AS (Autonomous System) path length, which counts how many different networks the request must traverse. The Tokyo server might be just two network hops away, while the Los Angeles server might be four hops, making Tokyo the automatic choice.

Interestingly, geographic distance doesn’t directly matter. A query from New Jersey might reach a server in Chicago instead of a physically closer New York if the network path to Chicago crosses fewer network boundaries due to better peering relationships between the ISPs involved.

The DNS root servers provide a perfect example of anycast in action. The historical limitation of 13 root server identities came from the original DNS specification requiring responses to fit in 512-byte packets, which could only accommodate 13 server entries with their names and IP addresses.

Today, each of these 13 identities represents dozens or even hundreds of physical servers. The F root server, operated by the Internet Systems Consortium, has over 150 locations worldwide, all responding to the same IP address 192.5.5.241. This distribution provides natural protection against DDoS attacks since overwhelming one location doesn’t affect the others, and the anycast routing automatically directs traffic away from attacked or failed locations.

Anycast also works particularly well for DNS queries because they are typically single-packet exchanges over UDP. A query arrives, the server responds, and the transaction completes without maintaining any connection state. This stateless nature means it doesn’t matter if subsequent queries from the same user go to different servers.

However, this approach has limitations. TCP connections, which maintain an ongoing state between packets, become complicated with anycast. If internet routing changes mid-connection, packets might suddenly start going to a different server that has no knowledge of the existing connection. Debugging also becomes challenging, as administrators need ways to identify which specific server handled a request when hundreds share the same IP address. Many anycast services include special headers or diagnostic endpoints that reveal the actual server location for troubleshooting purposes.

Despite these challenges, anycast has become a fundamental aspect of modern internet infrastructure.

Summary

In this article, we’ve looked at DNS, along with a detailed walkthrough of the DNS Records and the role of Anycast.

Here are the key learning points in brief:

  • DNS translates human-readable domain names like google.com into IP addresses that computers use to communicate, processing billions of requests per second globally through a massive distributed system.

  • When entering a domain in a browser, the query first checks multiple local caches (browser cache, OS cache, hosts file) before contacting external DNS servers, which is why frequently visited sites load faster.

  • The recursive resolver, typically provided by ISPs or public services like Google, tracks down IP addresses by querying multiple DNS servers on behalf of users.

  • DNS follows a strict hierarchy starting with 13 root servers (A through M) that direct queries to appropriate TLD servers, which then point to authoritative name servers containing the actual domain records.

  • The journey from root servers to TLD servers to authoritative servers creates a delegation chain, with each level knowing just enough to point to the next level down rather than storing everything.

  • Multiple layers of caching throughout the DNS system mean the first lookup of a domain might take 50-100 milliseconds, but subsequent queries return in under 10 milliseconds from cached responses.

  • A records map domains to IPv4 addresses while AAAA records handle IPv6 addresses, with websites often having multiple A records pointing to different geographic locations for basic load distribution.

  • CNAME records create aliases pointing one domain to another, commonly used for CDN integration, but cannot coexist with other record types for the same domain due to their fundamental design.

  • NS records establish the DNS delegation hierarchy, allowing organizations to split control of subdomains across different providers (like api.example.com on AWS while example.com stays on Cloudflare).

  • MX records handle email routing with priority values for failover, while TXT records store various text data, including domain verification codes and email authentication policies like SPF and DKIM.

  • TTL (Time To Live) values in seconds control how long DNS records are cached, with shorter TTLs enabling faster changes but increasing server load, while longer TTLs improve performance but slow propagation.

  • Anycast allows the same IP address to exist on hundreds of servers worldwide simultaneously, with internet routing protocols automatically directing queries to the nearest available server.

  • Anycast provides natural DDoS protection and automatic failover since attacking or losing one location doesn’t affect others, with traffic seamlessly redirecting to alternative servers.
