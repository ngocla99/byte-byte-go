A Guide to Retry Pattern in Distributed Systems

In a monolithic application, a function call is a local, in-memory process. Aside from a catastrophic hardware failure or a process crash, the execution of a function is essentially guaranteed. If the process is alive, the call succeeds.

However, in distributed systems, this guarantee does not hold. Components communicate over physical networks that are inherently unreliable. This reality is captured in the “Fallacies of Distributed Computing,” specifically the first fallacy: “The network is reliable”. In truth, it is not. A request sent from Service A to Service B may fail not because Service B is broken, but simply because the communication medium momentarily faltered.

This creates a need for defensive programming patterns, and one of the primary mechanisms we use is the Retry pattern. By automatically retrying a failed operation, a system can trade latency for availability, turning what would have been a failed user request into a successful one.

However, retries are both essential and dangerous in distributed systems. On the one hand, they transform unreliable networks into reliable ones. But on the other hand, indiscriminate retries can lead to latency amplification, resource exhaustion, and cascading failures that can take down entire platforms.

In this article, we will explore the retry pattern in depth, understand when and how to use it safely and effectively.

What is a Retry?

A retry is the practice of automatically reattempting a failed operation in the hope that a problem will resolve itself. This operation could be an API call, a database query, or any network request.

To understand why retries are necessary, let us consider what happens when Service A calls Service B over the network. The request does not travel directly. It can pass through load balancers, network switches, and possibly multiple data centers.

With so many potential failure points, errors are inevitable. At any point along this journey, packets can be lost, delayed, duplicated, or arrive out of order. The server might crash mid-response. A router might restart, or the DNS resolution might fail temporarily.

Retries provide multiple chances for the operations to succeed. The impact of these multiple chances can be substantial. According to a study by AWS, even a single retry can dramatically improve perceived availability. For example, a service with 99% availability jumps to 99.99% availability with just one retry.

A typical retry mechanism follows the steps below:

  • Initial Attempt: The application first tries to perform the operation, such as calling a service to retrieve data.

  • Detect Failure: If the operation fails, the retry logic kicks in.

  • Check the Limit: The system checks whether it has already reached its maximum number of retry attempts. This is a crucial safety measure to prevent infinite retry loops.

  • Wait and Retry: If the limit has not been reached, the system waits for a specific delay before trying the operation again.

  • Success or Final Failure: This cycle repeats until either the operation succeeds or the maximum attempts are exhausted. If all retries fail, the system treats the problem as a permanent error and handles it accordingly.

The Relation Between Retries and Timeouts

Retries and timeouts form a complementary pair.

Timeouts determine when to give up on an attempt or consider it a failure, while retries determine what to do after giving up. If a call does not complete within the specified time frame, it is timed out, and a retry may then be initiated according to the retry policy. This combination ensures that the system never waits indefinitely on a stuck operation. Instead, it fails fast and then retries.

When configuring timeouts, we need to think about the total time budget available. Consider a scenario where users will abandon after 30 seconds, and each attempt might take up to 10 seconds, including backoff delays. In this case, we can afford roughly 3 attempts.

So how do we choose the right timeout value?

Setting timeouts too short causes excessive retries and wasted work. However, setting them too long ties up resources and delays failure detection. AWS recommends basing timeouts on the latency percentiles of the downstream service. For example, if the service responds within 200ms for 99.9% of requests (p99.9 latency), setting the timeout at 200ms means only 0.1% of healthy requests will be incorrectly timed out.

In complex systems with deep call stacks, we can extend this concept through deadline propagation. Rather than each service having an independent timeout, the original request carries a deadline that propagates downstream. Each service subtracts its own processing time and passes the remaining budget to its dependencies. When the deadline expires, all pending work is canceled. This prevents wasted effort on requests that can no longer succeed anyway.

When Should You Retry?

Not all failures are created equal, and not all failures deserve a retry.

Retrying a request is based on the assumption that the failure is transient i.e. a temporary condition that is likely to resolve itself before the next attempt. If the failure is persistent or permanent, retrying is not only futile but harmful. It wastes computational resources and degrades system throughput.

Understanding the difference between transient and permanent failures is essential for implementing effective retry logic

Transient Failures

Transient failures stem from temporary conditions in the system. For example, in cloud-native environments like AWS or Azure, infrastructure is ephemeral. Virtual machines are migrated, and load balancers shift traffic. These events can generate transient connection errors. The system architecture must anticipate these and retry appropriately. Otherwise, the perceived availability of the application will drop significantly.

In other words, transient failures are often self-correcting anomalies that resolve within a short time. Some common examples include:

  • Network Glitches: Momentary loss of connectivity between two nodes, such as a TCP handshake timeout or a DNS resolution failure due to an update.

  • Service Unavailability: Service overload leading to rejected requests. This often manifests as HTTP 503 (Service Unavailable) or HTTP 429 (Too Many Requests). If the overload is bursty in nature, a retry after a short delay might find the service with available capacity.

  • Timeouts: A request takes longer than the configured threshold. This could happen due to a “stop-the-world” garbage collection pause on the server or a database lock.

For rate-limiting scenarios indicated by HTTP 429, we should respect the Retry-After header rather than using a fixed count. Server errors during deployments may need more patience than random network glitches.

Permanent Failures

Permanent failures represent structural or logical errors. No amount of retrying will fix the issue because the request is invalid or the system state prevents success. Here are some examples:

  • Client Errors (4XX Series): HTTP 400 (Bad Request) indicates malformed syntax. HTTP 401 (Unauthorized) indicates missing credentials. HTTP 404 (Not Found) indicates the resource does not exist. Retrying these is illogical. The result will always be the same, assuming the error codes are properly set up and not misleading.

  • Business Logic Failures: If a user attempts to withdraw money exceeding their balance, the transaction fails. Retrying will not increase their balance.

  • Security Blocks: HTTP 403 (Forbidden) implies a permanent lack of permission.

Retrying permanent errors is a common anti-pattern. It amplifies the load on the system without any possibility of success. For example, if a bug in the client code sends malformed JSON resulting in HTTP 400, and the client is configured to retry 5 times, the server must pay the processing cost to parse and reject the invalid payload 6 times instead of once.

The Gray Zone

Some errors can fall into an ambiguous category where the right course of action is not immediately clear. Here are some examples:

  • HTTP 500 (Internal Server Error): This is a generic catch-all status code. It might indicate a transient database connection issue, which is retriable, or a NullPointerException in the code, which is not retriable.

  • HTTP 502 (Bad Gateway) and HTTP 504 (Gateway Timeout): These usually indicate that an upstream dependency is failing. They are generally considered retriable since the upstream dependency might recover.

Retries in Synchronous and Asynchronous Situations

Synchronous retries occur within the same call flow that initiated the operation. For example, a user’s request to a web service triggers a call to a database, and if that call fails, the service retries it immediately while the user waits. Since the client expects an immediate result, we must choose the number of attempts and wait times carefully to balance success chances with user experience.

In interactive applications, it is often better to fail after a small number of quick retries and inform the user, rather than keep them waiting through many long delays.

Asynchronous retries decouple the retry process from the original request. If the client does not need the result immediately, we can offload the operation to a separate process or queue that handles retries over time.

Consider a user uploading a video that needs transcoding. The upload request enqueues a transcoding task to a worker. If the worker’s attempt fails due to a transient error, it can retry independently without making the user wait.

These mechanisms typically involve message queues where the failed operation is re-queued for retry after a delay. This allows many retry attempts spread over a longer period, even minutes or hours, since no human is actively waiting for the result.

Deciding The Number of Retries

When retrying, operation type matters significantly. For example, user-facing synchronous calls should retry 1 to 3 times because users will not wait beyond a few seconds. However, background asynchronous jobs can tolerate 5 or more retries with longer backoff periods since latency matters less than eventual success.

One approach to this problem uses dual budgets: a per-request maximum of 3 attempts, combined with a per-client budget limiting retries to 10% of total requests. This means if most requests are failing, the client stops retrying entirely to avoid amplifying the problem.

How to Retry?

Once the decision to retry is made, the next critical architectural decisions are timing and frequency.

How many times should we retry? And how long should we wait between attempts?

A retry mechanism must include a limit on the number of attempts. Unlimited retries can lead to a vicious cycle and resource exhaustion. Typically, the right number is tied to how long a user is willing to wait.

The strategy we choose determines the system’s resilience. A poor strategy can transform a minor hiccup into a major outage through synchronization effects. Let us examine the common approaches.

1 - Immediate Retry

The simplest strategy is to retry immediately upon receiving an error.

This is rarely the correct approach in distributed systems. If the failure is due to load (HTTP 503 or 429), immediate retries can act as a denial-of-service attack against our own server. The server, already struggling to process work, is bombarded with retry requests instantly. It consumes CPU cycles for connection management and request rejection, preventing it from recovering.

2- Fixed Interval Backoff

Backoff means intentionally waiting between retry attempts. It is essential because immediate retries hammer an already struggling system. If a service fails due to overload, retrying instantly adds more load at the worst possible moment. Backoff gives the system breathing room to recover.

With fixed interval backoff, the client waits a constant duration, say 200 milliseconds, between each attempt.

While better than immediate retries, fixed intervals are rigid. They do not adapt to the severity of the failure. If a database is rebooting and takes 30 seconds, retrying every 200 milliseconds results in 150 futile attempts, wasting bandwidth and thread resources.

Furthermore, fixed intervals promote synchronization. If 100 clients fail simultaneously, they all retry simultaneously 1 second later, creating synchronized waves of load that prevent recovery.

3 - Linear Backoff

Linear backoff increases delays by a constant amount with each attempt: 1 second, then 2 seconds, then 3 seconds, then 4 seconds. This spreads retries over time more effectively than fixed delays. After 5 retries, the total wait time is 15 seconds.

Linear backoff works well for lightweight operations where predictable timing matters.

4 - Exponential Backoff

Exponential backoff is the industry standard. In this approach, the wait time increases exponentially with each failed attempt. For example, wait 1 second, then 2 seconds, then 4 seconds, then 8 seconds.

This strategy aligns with the nature of recovery. If a service fails, it often needs time to stabilize, perhaps for a leader election in a cluster or a TCP slow-start. By backing off aggressively, the client reduces the load on the failing service, giving it breathing room to recover.

When implementing exponential backoff, we typically configure an initial wait time (the base) and a cap (maximum delay) to prevent the wait time from growing into minutes, which would degrade user experience unacceptably.

5 - The Thundering Herd Problem and the Necessity of Jitter

Using exponential backoff alone has a fatal flaw known as correlation. Consider a scenario where a microservice relies on a database.

  • The database experiences a momentary lock, causing 1,000 requests to fail at time T = 0.

  • If all 1,000 clients use the same exponential backoff algorithm, they will all return to retry at exactly T = 1 second.

  • The database, just recovering, is hit by 1,000 simultaneous requests. It fails again.

  • The clients all wait 2 seconds and return together at T = 3 seconds.

This phenomenon is known as the Thundering Herd problem or Request Synchronization. It creates spikes of load separated by idle periods, which is the worst possible traffic pattern for a recovering server.

The solution is to use jitter.

Jitter introduces randomness into the wait time. By adding a random component to the backoff duration, the retries become desynchronized. Instead of 1,000 clients hitting at T = 1, they arrive spread out between T = 0.5 and T = 1.5. This smearing of traffic flattens the spike, converting a burst into a steady stream that the server can process more easily.

There are several jitter algorithms we can use:

  • Full Jitter picks a random delay between 0 and the calculated backoff.

  • Equal Jitter takes half the calculated backoff plus a random portion of the other half, ensuring some minimum delay while still spreading load.

  • Decorrelated Jitter bases each delay on the previous random value, creating varied but connected timing.

For the majority of distributed system implementations, exponential backoff with full jitter is the optimal strategy. It balances the need to back off (to help the server recover) with the need to retry quickly (to minimize client latency).

The Risks of Retrying

While retries handle local failures, they also introduce risks. In a complex microservices architecture, the interaction between retries at different layers can lead to catastrophic failure modes.

Let us examine the primary risks.

1 - Retry Amplification

The most critical problem is retry amplification across service layers.

In microservice architectures, a single user request might traverse multiple services. If each layer implements retries independently, the multiplication effect becomes problematic. This exponential growth, expressed as K^N, where K is the number of retry attempts and N is the service depth, explains how seemingly reasonable retry policies can cause system meltdowns.

The best practice is to retry at a single point in the stack, typically at the outermost layer closest to the user.

2 - Latency Amplification

Retries inherently extend the duration of a request. If a service has a timeout of 2 seconds and configures 3 retries, the maximum latency for that call becomes:

MaxLatency = Timeout × (1 + Retries) + BackoffDelays

Consider Service A calling Service B, which in turn calls Service C. At each hop, the potential latency multiplies.

What started as a 2-second timeout can balloon into something much higher by the time backoff delays are factored in. Users experience this as unpredictable response times, and upstream services may themselves timeout waiting for responses.

3 - Resource Exhaustion

Every request that is waiting to retry holds onto resources.

In synchronous I/O models like thread-per-request in traditional Java and Tomcat applications, the thread is blocked during the sleep or network wait.

Aggressive retries consume finite resources rapidly. Thread pools are utilized with blocked requests. Connection pools are exhausted and memory fills up with queued work.

How to Make Retries Safe?

A retry is safe when repeating it produces no additional side effects beyond the original operation. Here are some strategies to manage this:

1 - Idempotency

The greatest functional risk of retrying is executing an operation twice. In banking, e-commerce, or inventory management, this is unacceptable.

See the diagram below:

Consider the scenario where a client sends a request to pay $100. The server processes it, but the network fails while sending the acknowledgment. The client times out and retries. The server receives the payment request again. Without protection, the user is charged $200.

The solution is idempotency. An operation is idempotent if executing it multiple times yields the same result as executing it once.

Some operations are idempotent by nature:

  • Reading data (HTTP GET) can be repeated indefinitely since we get the same information each time.

  • Deleting a resource (HTTP DELETE) is idempotent because deleting something twice leaves it equally deleted.

  • Setting a value to a specific state (HTTP PUT with complete replacement) is idempotent because multiple identical updates produce identical results.

As can be seen, creating a resource (HTTP POST) might result in duplicates on retry. For example, charging a credit card, sending an email, or transferring money must happen exactly once, yet network failures make it impossible to know whether the server received and processed the original request. The danger zone is the unknown response: the client sends a request, waits, times out, but never receives a response.

Did the server process the request before crashing? Did the network lose the response? Without additional mechanisms, retrying risks duplicate execution.

The Idempotency Key Pattern

To make non-idempotent operations safe, we use an idempotency key.

The pattern works as follows:

  • The client generates a unique key, typically a UUID, for each business operation and includes it with the request.

  • The server stores this key along with the operation’s result.

  • When a retry arrives with the same key, the server returns the cached result instead of re-executing the operation.

The server’s implementation of this contract is critical. Recording the client’s token and executing the state-changing operation must be atomic. This requires the operation to meet ACID properties (Atomicity, Consistency, Isolation, Durability) to prevent inconsistent states where the token is recorded but the operation fails, or vice versa.

When the server receives a request with a token it has already processed, it must return a semantically equivalent response. This is the same outcome the client would have received if the initial request had succeeded. This transparency allows client-side SDKs to handle retries automatically without changing their logic based on an “already exists” error.

Alternative Approaches to Idempotency

Beyond idempotency keys, there are other ways to achieve safe retries:

  • Database Deduplication: Use the request ID as a unique key in a database. When processing a payment request, insert a record with the idempotency key. If the same key is inserted again, the database rejects the duplicate through a unique constraint violation, and the application can fetch the existing transaction instead of creating a new one.

  • Server-Side Fingerprinting: For legacy systems where clients cannot be modified to send unique IDs, the server can create a fingerprint of the request payload, such as a hash of the important fields. However, this approach is less reliable. It struggles with requests containing timestamps or unique values that change on each retry, and it can misidentify two distinct operations as duplicates if they have identical data but different intent.

Explicit idempotency keys remain the preferred solution in modern systems.

2 - Server-Side Retry Budgets

While exponential backoff helps, it is a local decision made by each client independently. The server needs a global defense mechanism. And hence the role of retry budgets.

The concept is simple: limit retries not by a fixed count, but by a ratio relative to successful requests. The commonly used rule is 10%. In this approach, we track the ratio of retry attempts to total requests over a rolling window and only issue a retry if this ratio remains below 10%.

This creates self-regulating behavior. When most requests are succeeding during normal operation, retries happen freely. When failures are widespread and the system is under stress, the budget quickly exhausts, thereby halting retries. This caps the worst-case amplification.

For reference, Google implements this at two levels: per-request (maximum 3 attempts) and per-client (10% retry ratio). Google gRPC also provides built-in support through retry throttling configuration that uses a token bucket approach. Clients earn tokens on successful requests and spend them on retries.

3 - Circuit Breaker Pattern

Retries handle transient faults. On the other hand, circuit breakers handle persistent faults. They are complementary patterns.

Inspired by electrical circuit breakers that trip to prevent fires, the circuit breaker pattern wraps remote calls and monitors for failures. The circuit breaker operates in three states:

  • Closed State: The circuit operates normally, allowing requests to pass through. The breaker monitors failure rates.

  • Open State: When failures exceed a threshold, the circuit trips open. All requests fail immediately without attempting the remote call. This gives the failing service time to recover.

  • Half-Open State: After a timeout period, the circuit allows a limited number of test requests through. If these succeed, the circuit closes and normal operation resumes. If they fail, the circuit opens again.

The retry logic should sit outside the circuit breaker. However, if the circuit breaker is open, the retry logic should observe this and stop retrying. There is no point in backing off and retrying if the circuit is open because the call will just fail locally anyway.

The circuit breaker complements retry logic by stopping retries entirely when a service is known to be down. Rather than waiting for each request to timeout and retry repeatedly, the system fails fast with a clear signal that recovery is underway.

Summary

In this article, we’ve looked at retries in great detail. Here are the key learning points in brief:

  • Retries are a fundamental resilience technique in distributed systems that transform unreliable networks into reliable ones by automatically reattempting failed operations, but they must be implemented carefully to avoid causing more harm than good.

  • Not all failures deserve a retry. Transient failures like network glitches, temporary service unavailability, and timeouts are good candidates for retries, while permanent failures like client errors (4XX) and business logic failures should not be retried.

  • Timeouts and retries work as a complementary pair where timeouts determine when to give up on an attempt, and retries determine what to do after giving up. The total time budget available to the user should guide both configurations.

  • Exponential backoff with full jitter is the industry standard retry strategy. Exponential backoff gives failing services breathing room to recover, while jitter introduces randomness that prevents the thundering herd problem where synchronized retries overwhelm recovering servers.

  • Retries introduce systemic risks, including retry amplification across service layers, latency amplification that extends request duration unpredictably, and resource exhaustion that can starve thread pools and connection pools.

  • Idempotency is essential for safe retries. Operations must produce the same result whether executed once or multiple times, and idempotency keys allow non-idempotent operations like payments to be safely retried without duplicate execution.

  • Server-side retry budgets and circuit breakers act as global safety mechanisms. Retry budgets limit retries to a percentage of total traffic, while circuit breakers stop retries entirely when a service is known to be down, allowing it time to recover.

  • For failures that persist beyond all retry attempts, dead letter queues capture failed messages for later analysis and reprocessing, ensuring that no work is silently lost.
