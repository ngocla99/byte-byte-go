A Guide to Database Sharding

When applications grow popular, they often face a good problem of attracting more users, and exponentially more data. While this growth signals business success, it creates technical challenges that can cripple even well-designed systems. The database, often the heart of any application, becomes the bottleneck that threatens to slow everything down.

Unlike application servers, which can be easily scaled to handle more traffic, databases resist horizontal scaling. We cannot simply add more database servers and expect our problems to vanish. This is where sharding enters the picture as an important solution to one of the most persistent challenges in modern application architecture.

In this article, we will learn about database sharding in more detail. We will understand what it is, why it matters, how different approaches work, and what key considerations are important when implementing it.

What Is Database Sharding?

Database sharding is an architectural pattern that splits a large database into smaller, more manageable pieces called shards. Think of it as dividing a massive library into multiple buildings, each housing a specific collection of books. When someone needs a book, they only need to search one building rather than the entire collection.

Sharding builds on the foundation of horizontal partitioning. In horizontal partitioning, we divide a table’s rows into multiple smaller tables based on a partition key. These smaller tables, called partitions, reduce the effort required to query and manipulate data. However, traditional partitioning keeps all these partitions on the same computer.

Sharding takes this concept further by distributing these partitions across different computers or nodes. This distribution enables better scalability and performance by harnessing the combined resources of multiple machines working in parallel.

Different database systems use different terminology for these concepts. MongoDB calls its partitions shards. Couchbase uses the term vBucket. Apache Cassandra refers to them as vNodes. Despite the varied naming, the fundamental concept remains the same, which is dividing data into smaller units to improve performance and scalability.

The Benefits of Sharding

The benefits of sharding fall into three main categories:

  • Scalability and Growth: When we distribute a large dataset across multiple shards, we spread the query load across multiple nodes. Each node independently executes queries for its assigned data. More importantly, we can add new shards dynamically at runtime without shutting down the application for maintenance.

  • Performance Improvements: Retrieving data from a single massive database is time-consuming. The query must search through millions or billions of rows to locate the required information. In contrast, shards contain only a subset of the total rows. This reduced search space translates to faster data retrieval because queries process fewer rows.

  • Enhanced Availability: In a traditional single-server database architecture, if the hosting node fails, the entire dependent application experiences downtime. Sharding mitigates this risk by distributing data across multiple nodes. When one node fails, the application continues operating using the remaining shards.

Sharding and Replication

Sharding rarely works alone in production systems. Organizations typically combine it with replication to achieve high availability and fault tolerance.

For reference, replication involves creating multiple copies of data and storing them on different nodes. In a leader-follower replication model, one node acts as the leader handling write operations, while follower nodes replicate the leader’s data and handle read operations.

By replicating each shard across multiple nodes, the system ensures data remains accessible even when individual nodes fail. A single node can store multiple shards, each of which may serve as either a leader or a follower. This distributed architecture maintains data availability and resilience despite node failures or network disruptions.

Sharding Strategies

The primary objective of database sharding is to evenly distribute data and query load across multiple nodes. However, if data partitioning is unbalanced, some shards may handle disproportionately more data or queries than others. This scenario, called a skewed shard, diminishes sharding benefits. In extreme cases, a poorly designed strategy results in a hot spot where one node becomes overwhelmed while others remain idle.

Selecting an appropriate sharding strategy is crucial for avoiding these problems. Let’s examine the commonly used approaches.

Range-Based Sharding

Range-based sharding splits database rows based on value ranges.

Each shard receives a continuous range of keys spanning from minimum to maximum values. Keys within each shard are maintained in sorted order to enable efficient range scans. For example, consider a product database storing product information. Range-based sharding might split the database into shards based on price ranges. One shard could store all products priced between $0 and $75, while another contains products priced between $76 and $150.

However, range-based sharding has a significant drawback. Certain access patterns can create hot spots. If most products fall within a specific price range, the shard storing that range experiences disproportionately high load while other shards remain underutilized.

Key or Hash-Based Sharding

Hash-based sharding assigns keys to shards using a hash function. A well-designed hash function plays a crucial role in achieving balanced distribution. Instead of assigning key ranges to shards, this approach assigns hash ranges to shards. Consistent hashing is one technique often used to implement this strategy.

The main advantage of hash-based sharding is its ability to distribute keys fairly among shards. However, it comes with a trade-off.

Using the hash of the key instead of the key itself means we lose the ability to perform efficient range queries. Adjacent keys may scatter across different partitions, and their natural sort order is lost. Hash-based sharding reduces hot spots but cannot eliminate them. For example, on a social media website, a celebrity user posting content can generate a massive volume of writes to the same key.

Directory-Based Sharding

Directory-based sharding relies on a lookup table to determine how records are distributed across shards. The lookup table serves as a directory mapping the relationship between data and the specific shard where it resides.

The main advantage is flexibility compared to other strategies. It allows greater control over data placement across shards because the mapping is explicitly defined. However, this approach heavily relies on the lookup table. Any issues or failures related to it can impact overall database performance and availability.

Geographic or Geo-Sharding

Geographic sharding distributes data based on the physical location of users or resources. This approach places data closer to where it will be accessed most frequently, reducing latency and improving user experience.

For example, a global social network might store European user data in data centers located in Europe, while American user data resides in North American data centers. Geo-sharding offers significant advantages for applications serving a global user base and helps comply with data sovereignty regulations. However, it introduces challenges when users need to access data across geographic boundaries.

Selecting the Right Shard Key

Choosing an appropriate shard key is crucial for implementing an effective sharding strategy.

Database designers should consider several key factors when making this decision:

  • Cardinality refers to the number of possible values a shard key can have. It determines the maximum number of shards that can be created. For example, if we choose a Boolean field as the shard key, the system will be limited to only two shards. To maximize horizontal scaling benefits, selecting a shard key with high cardinality is generally recommended.

  • Frequency represents how often a particular shard key value appears in the dataset. If a significant portion of records contains only a subset of possible shard key values, the shard storing that subset may become a hotspot. For instance, if a fitness website’s database uses age as the shard key, most records may end up in the shard containing subscribers between ages 30 and 45, leading to uneven data distribution.

  • Monotonic change refers to shard key values that increase or decrease over time for a given record. If a shard key is based on a monotonically changing value, it can result in unbalanced shards. Consider a database storing user comments where Shard A stores users with fewer than 10 comments, Shard B stores users with 11 to 20 comments, and Shard C stores users with more than 30 comments. As users continue adding comments, they gradually migrate to Shard C, making it increasingly unbalanced.

The Challenges of Sharding

While sharding offers powerful benefits, it introduces complexity to database systems.

One of the most significant challenges is executing queries that span multiple shards. When data resides on a single server, joining tables or aggregating data is straightforward. In a sharded environment, the database must coordinate across multiple nodes to execute such queries, adding latency and complexity.

Maintaining ACID properties across shards presents another major challenge. In a traditional database, transactions ensure that operations either all succeed or all fail. In a sharded system, implementing distributed transactions introduces performance overhead. Many sharded systems accept eventual consistency instead of strict ACID guarantees to maintain performance.

Sharding often requires changes to application code. Applications may need to become shard-aware, understanding which shard contains specific data. This awareness complicates the codebase and makes it harder to maintain. Operations that were simple in a single-database system, such as generating unique IDs or maintaining referential integrity, become complex in sharded environments. For example, auto-incrementing primary keys that work in a single database require coordination mechanisms in sharded systems. Some solutions include using UUIDs, generating IDs based on shard identifiers combined with local sequences, or employing centralized ID generation services. However, each approach has trade-offs in terms of performance, uniqueness guarantees, and implementation complexity.

Rebalancing Shards Over Time

Databases undergo various changes over time that necessitate redistributing data and workload across nodes. Increased query throughput requires additional CPU resources. Dataset growth demands more storage capacity. Machine failures result in other machines assuming their responsibilities. This process is known as shard rebalancing.

In a fixed partition configuration, the number of shards is determined when the database is initially set up and typically remains unchanged. For instance, consider a database running on a cluster of 10 nodes split into 100 shards. If a new node joins the cluster, the system can rebalance by migrating a few shards from existing nodes to the new node. In this approach, only entire shards move between nodes during rebalancing.

Dynamic sharding adjusts the number of shards based on the total data volume stored in the database. When the dataset is relatively small, dynamic sharding maintains a low number of shards. As the dataset grows, it automatically increases the number of shards to accommodate expanded data volume. One key advantage of dynamic sharding is its flexibility, as it can be applied to both range-based and hash-based sharding strategies.

Request Routing in Sharded Databases

After sharding the dataset, determining how a client knows which node to connect to for incoming requests becomes critical. This problem is challenging because shards can be rebalanced and shard-to-node assignments can change dynamically.

There are three main approaches to address this challenge:

  • Shard-Aware Node: Clients contact any node using a round-robin load balancer. If the contacted node owns the shard relevant to the request, it handles it directly. Otherwise, it forwards the request to the appropriate shard.

  • Routing Tier: Client requests are sent to a dedicated routing tier that determines the node responsible for handling each request and forwards it appropriately.

  • Shard-Aware Client: Clients are aware of shard distribution across nodes. The client first contacts a configuration server to obtain the current shard-to-node mapping, then directly connects to the appropriate node.

Different databases employ different mechanisms to keep routing information updated. For example, LinkedIn’s Espresso uses Helix for cluster management, which relies on ZooKeeper to store configuration information. MongoDB utilizes a separate configuration server and a dedicated routing tier. Apache Cassandra employs a gossip protocol among nodes to propagate changes to the mapping.

Real-World Implementations

Understanding how major companies implement sharding provides valuable insights into practical considerations. Here are some examples:

  • Instagram uses PostgreSQL with custom sharding logic to handle billions of user records. They partition user data based on user IDs using consistent hashing, allowing them to scale horizontally while maintaining the benefits of a relational database. Their sharding implementation allows them to add new database servers without massive data migrations. Each PostgreSQL instance holds multiple logical shards, and Instagram’s application layer maintains shard mapping to determine which database instance to query. This approach has proven remarkably effective, allowing Instagram to grow from millions to billions of users while maintaining query performance.

  • MongoDB provides built-in sharding capabilities that abstract complexity from application developers. It uses config servers to track metadata about which chunks of data reside on which shards. The mongos routing service acts as an intermediary between applications and shard servers, directing queries to the appropriate shards. This architecture allows applications to interact with a sharded cluster as if it were a single database.

  • Discord built its messaging infrastructure on top of Apache Cassandra, choosing hash-based sharding for message data. They found that range-based sharding created hot spots during periods of high activity in specific channels, while hash-based sharding distributed load more evenly across their cluster. However, this came at the cost of making it more difficult to retrieve consecutive messages efficiently.

  • YouTube developed Vitess, a database clustering system for horizontal scaling of MySQL. Vitess acts as a proxy layer that handles sharding, connection pooling, and query routing. It allows YouTube to use MySQL while achieving massive scale through sharding. This system has become so successful that it is now available as an open-source project used by many other companies.

When Not to Use Sharding

Understanding when to avoid sharding is as important as knowing when to use it.

Before implementing sharding, we should exhaust simpler scaling options. Vertical scaling, adding more CPU and memory to existing servers, is often sufficient for many applications. Query optimization, proper indexing, and database tuning can dramatically improve performance without the complexity of sharding.

Sharding is unsuitable for applications with complex joins and transactions. If the application frequently needs to join data across what would become different shards, sharding will significantly degrade performance. Small to medium-sized applications rarely need sharding. If the database fits comfortably on a single powerful server with room to grow, the added complexity of sharding outweighs its benefits.

Summary

In this article, we’ve looked at database sharding in detail. Here are the key learning points in brief:

  • Database sharding is a powerful technique for scaling databases horizontally, but it comes with complexity that should not be underestimated.

  • The choice of sharding strategy depends heavily on specific use cases, access patterns, and business requirements.

  • Range-based sharding works well when range queries are common.

  • Hash-based sharding provides better load distribution, but sacrifices range query efficiency.

  • Directory-based sharding offers flexibility at the cost of managing an additional lookup system

  • Geo-sharding excels for global applications with location-based access patterns.

  • When implementing sharding, we must carefully consider the shard key selection, taking into account cardinality, frequency, and monotonic change characteristics. We also need to plan for operational challenges like cross-shard queries, distributed transactions, shard rebalancing, and request routing.

  • Sharding is not always the answer to any database scaling challenge. Exhausting simpler scaling options first and understanding when sharding is truly necessary can save significant time and resources.
