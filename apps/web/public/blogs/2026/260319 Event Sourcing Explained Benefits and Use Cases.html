Event Sourcing Explained: Benefits and Use Cases

Every time we run an UPDATE statement in a database, something disappears. The old value, whatever was there a moment ago, is gone.

In fact, most databases are designed to forget. Every UPDATE overwrites what came before, every DELETE removes it entirely, and the application is left with only a snapshot of the present state. We accept this as normal because it’s the most natural way to think about things.

But what if your system needs to answer a different kind of question: not just “what is the current state?” but “how did we get here?”

That’s the question Event Sourcing is built to answer. And the solution is both more rewarding and more demanding than it first appears. In this article, we will look at Event Sourcing along with its benefits and trade-offs.

What CRUD Loses

In a traditional CRUD system, every write operation actually destroys information. We don’t usually think of it this way because the system does exactly what we asked: store the latest state. However, the latest state is the only thing we get back.

Consider an e-commerce order. A customer creates an order, adds three items, removes one, applies a coupon code, removes it, then applies a different coupon. In a CRUD database, the orders table holds a single row reflecting the final state of that order, including the current items, the current total, and the current coupon.

But what if the customer calls support claiming they were charged incorrectly? What if a product manager wants to understand why so many users remove items before checkout? What if an auditor needs to know whether the first coupon was ever applied at all?

The database can’t answer any of these questions. That history was overwritten with each update.

For many applications, this is entirely fine. A user profile page doesn’t need to remember every previous email address. But for systems where the journey matters as much as the destination, CRUD’s habit of forgetting becomes a real limitation.

Event Sourcing starts from a different premise entirely.

How Event Sourcing Works

Instead of storing the current state of your data and updating it in place, Event Sourcing stores every change as an immutable event. The current state isn’t saved directly. It’s derived by replaying those events in order.

Take the same e-commerce order from before. In an event-sourced system, the storage looks completely different. Rather than a single row that gets updated, you’d see a sequence of events such as:

  • OrderCreated

  • ItemAdded (Product A)

  • ItemAdded (Product B)

  • ItemAdded (Product C)

  • ItemRemoved (Product B)

  • CouponApplied (SAVE20)

  • CouponRemoved (SAVE20)

  • CouponApplied (WELCOME10)

Each event is a fact, a record of something that happened. Events are immutable, meaning once they’re written, they never change. They’re stored in an append-only log called an event store. No updates, no deletes. New events only get added to the end.

To determine the current state of the order, the system replays these events from start to finish and computes the result. The final state is the same as what a CRUD system would have stored, but now we also have the complete story of how it got there.

This opens up capabilities that traditional persistence simply can’t offer:

  • A complete audit trail: Every change is recorded, timestamped, and preserved. We know exactly what happened and when, without building a separate logging infrastructure

  • Time travel: We can reconstruct the state of any entity at any point in time by replaying events up to that moment.

  • Replayability: Since the event log is the source of truth, we can reprocess it to build entirely new views of your data. If we realize months later that we need analytics on coupon usage patterns, the data is already there.

One important clarification before we go further is that Event Sourcing is not the same thing as Event-Driven Architecture.

Event-Driven Architecture is a communication pattern where system components talk to each other through events. On the other hand, Event Sourcing is a storage pattern or a way of persisting data. We can use one without the other.

There’s also a subtle long-term challenge worth knowing about. Events are immutable, but business requirements change. What happens when we need to restructure how an event captures a concept, but we can’t modify the historical events already in the store?

This is the event versioning problem, and teams that adopt Event Sourcing consistently cite it as one of the trickiest challenges they face over time. Strategies like event upcasting (transforming old event formats into new ones during replay) and versioned schemas exist to handle this, but it’s something to plan for early rather than discover later.

The Querying Problem and Where CQRS Fits In

There’s a practical problem that surfaces almost immediately when you try to build a real application on top of an event store.

An event store is optimized for appending and reading back a sequence of events for a specific entity. It’s not designed to answer questions across entities.

“Show me all orders over $100 from the past week” is a trivial query in a relational database with a CRUD model. In an event store, answering that question would mean replaying every order’s entire event history, computing each order’s current state, and then filtering the results. That approach does not scale.

This isn’t a minor inconvenience. Nearly every real application needs to query data in ways that an event store simply can’t support efficiently. And this is where CQRS, Command Query Responsibility Segregation, enters the picture.

CQRS separates your system into two sides:

  • The write side handles commands and appends events to the event store.

  • The read side maintains separate, query-optimized views of your data called projections, sometimes also called read models.

A projection is a background process that consumes the event stream, processes each event, and writes the result into whatever database format works best for your queries. We might have a projection that maintains a relational table of current order summaries, another that feeds a search index, and another that powers a dashboard. Each one is built from the same event stream but shaped for a different use case.

The tradeoff this introduces is eventual consistency. Since projections are updated asynchronously, the read side lags behind the write side. For example, a customer places an order, but the order summary view might take a moment to reflect it. For many systems, this delay is imperceptible and perfectly acceptable. For others, it’s something we need to design around carefully.

CQRS isn’t a separate pattern you might optionally “add on” to Event Sourcing. It’s a near-necessity. The querying limitation of event stores makes some form of read-side separation almost mandatory for any production system.

The write side has its own performance concern as well. Replaying hundreds of thousands of events to reconstruct an entity’s state on every write gets slow. The common solution is snapshotting, which means periodically saving the computed state at a point in time so the system only needs to replay events since the last snapshot rather than the entire history. Between projections on the read side and snapshots on the write side, a production event-sourced system almost always involves more components than “just store events” suggests.

When It’s Worth the Tradeoff

Event Sourcing combined with CQRS is a powerful data architecture. However, adopting this pattern where it doesn’t belong creates complexity without payoff.

Event Sourcing changes how you model, persist, query, debug, and evolve your data. This isn’t a one-time setup cost you pay at the start of a project, but an ongoing constraint on every design decision you make going forward.

The learning curve is steep because it requires a fundamentally different way of thinking about data, one that’s unfamiliar to most developers coming from a CRUD background. And eventual consistency, an inherent consequence of separating reads and writes, means you need to design your entire user experience around the reality that reads may lag writes.

So when does the trade make sense?

Event Sourcing fits naturally in systems where the history of changes carries genuine business value. Financial systems are the canonical example, and for good reason. Accounting is already event-sourced in principle. Every credit and debit is recorded, and the balance is a derived calculation. The pattern also shines in order management systems where tracking the full lifecycle of an order matters for dispute resolution and analytics, in regulated industries where you need to explain exactly how the system arrived at a given state, and in collaborative tools where multiple users make changes to a shared state and the sequence of those changes matters.

On the other hand, the pattern adds unnecessary weight to simple CRUD applications, content management systems, and user settings pages. Anywhere you’re storing and retrieving the current state with no business need for history, traditional persistence does the job with far less overhead.

The most pragmatic advice from practitioners who’ve spent years working with Event Sourcing is to apply it selectively. Use it for the core domain where history matters, and use traditional CRUD for supporting functionality where it doesn’t. Event Sourcing is a module-level decision, not an all-or-nothing system architecture. A single system can have event-sourced components alongside conventional CRUD components. And often, that hybrid approach is exactly the right call.

Summary

In this post, we’ve looked at event sourcing in detail. Here are the key learning points in brief:

  • Traditional CRUD persistence overwrites state on every update, which means the history of how your data arrived at its current form is lost permanently.

  • Event Sourcing flips this by storing every change as an immutable event in an append-only log. The current state is derived by replaying these events in order rather than being stored directly.

  • The event log gives you a complete audit trail, the ability to reconstruct state at any point in time, and the option to reprocess events to build entirely new views of your data.

  • Event Sourcing is a storage pattern, not a communication pattern. It is not the same thing as Event-Driven Architecture, and confusing the two leads to real architectural problems.

  • An event store is not designed for efficient querying across entities, which makes CQRS (separating the read and write sides of your system) a near-necessity in any production implementation.

  • CQRS introduces eventual consistency as an inherent tradeoff. The read side will always lag slightly behind the write side, and your system needs to be designed with this in mind.

  • Event Sourcing fits naturally in domains where the history of changes has genuine business value, such as financial systems, order management, regulated industries, and collaborative tools.

  • The most pragmatic approach is to apply Event Sourcing selectively to your core domain and use traditional CRUD for supporting functionality. It’s a module-level decision, not an all-or-nothing system architecture.
