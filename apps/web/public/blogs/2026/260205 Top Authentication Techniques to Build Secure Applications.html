Top Authentication Techniques to Build Secure Applications

Authentication serves as the first line of defense in ensuring the security of applications and the sensitive data they handle. Whether it’s a personal banking app, a corporate platform, or an e-commerce website, effective authentication mechanisms are needed to verify the identity of users and safeguard their access to resources.

Without proper authentication, applications are vulnerable to unauthorized access, data breaches, and malicious attacks, potentially resulting in significant financial loss, reputational damage, and privacy violations.

In addition to security, authentication plays a critical role in the user experience. By identifying users, applications can provide personalized services, remember user preferences, and enable functionalities like Single Sign-On (SSO) across platforms.

With evolving threats, implementing secure and efficient authentication is more challenging than ever. Developers must navigate between competing priorities such as security (ensuring protection against different attack types like session hijacking, token theft, and replay attacks), scalability (supporting millions of users without compromising performance), and user experience (maintaining ease of use while applying strong security measures).

To tackle these challenges, developers rely on various authentication techniques. In this article, we’ll explore multiple authentication techniques used in applications and understand their advantages and disadvantages.

Fundamentals of Authentication

Authentication is the process of verifying the identity of a user, device, or system attempting to access an application. In simpler terms, it is how an application ensures that a person or system is who they claim to be. It often involves validating credentials such as usernames, passwords, biometric data, or tokens.

For example, when we log in to a website using a password, the application compares our input with stored credentials to confirm our identity. In API-based systems, applications might use tokens to verify that the calling service has the right to interact with the backend.

Authentication vs Authorization

Authentication and authorization are closely related but distinct processes.

Authentication answers the question “Who are you?” and focuses on verifying identity. Authorization answers the question “What are you allowed to do?” by determining the permissions or access levels granted to an authenticated user.

For example, authentication verifies that we are a registered user of an e-commerce platform. Authorization determines whether we can view the order history or manage the inventory as an admin. In other words, while authentication establishes identity, authorization enforces access control based on that identity.

Authentication with Cookies and Sessions

We will now look at the use of cookies and sessions concerning authentication:

Cookies

Cookies are small pieces of data stored on the client’s browser by a web server. They play a key role in maintaining state in otherwise stateless HTTP communication by enabling web applications to remember information across multiple requests.

Cookies allow web servers to store data that persists between requests, making them useful for various purposes such as session management (tracking logged-in users with session IDs), personalization (storing user preferences or settings), and analytics and tracking (recording user behavior for analytics or targeted advertising).

In authentication, cookies are commonly used to store session tokens or identifiers that validate a user’s identity.

How Cookies Work in Authentication

The process typically involves two parts. During user login, the user provides credentials like username and password. The server authenticates the credentials and generates a session ID or token. This session ID is sent to the browser and stored in a cookie.

For subsequent requests, the browser automatically includes the cookie in the Cookie header of each HTTP request. The server reads the session ID from the cookie, verifies it, and identifies the user.

Cookie Attributes

Cookies come with several attributes that control their behavior and security:

  • The HttpOnly attribute prevents JavaScript from accessing the cookie, mitigating Cross-Site Scripting (XSS) attacks. The Secure attribute ensures cookies are only sent over HTTPS connections, protecting against man-in-the-middle attacks.

  • The SameSite attribute helps prevent Cross-Site Request Forgery (CSRF) attacks. It has three values: Strict (cookie is never sent in cross-site requests), Lax (cookie is sent only with top-level navigations), and None (cookie is sent with all requests, but requires the Secure attribute).

  • The Domain attribute specifies which hosts can receive the cookie. For example, setting Domain=example.com allows the cookie to be sent to example.com and all its subdomains. The Path attribute restricts the cookie to specific URL paths.

  • For expiration, we can use Max-Age (specifies lifetime in seconds) or Expires (sets an absolute expiration date). If neither is set, the cookie becomes a session cookie and is deleted when the browser closes.

First-Party vs Third-Party Cookies

First-party cookies are set by the domain we are currently visiting.

When we visit example[dot]com, and it sets a cookie, that’s a first-party cookie. These cookies are used for essential site functionality like authentication and user preferences.

Third-party cookies are set by a different domain than the one we are visiting. For instance, if we visit example[dot]com and it loads an advertisement from ads.com, the ads.com domain can set a cookie. These cookies are commonly used for tracking user behavior across multiple websites for advertising purposes.

Modern browsers increasingly restrict third-party cookies due to privacy concerns. Some browsers block them by default. For authentication purposes, we should always use first-party cookies to ensure they work reliably across all browsers.

Challenges of Using Cookies

Cookies face several security challenges:

  • If a malicious script is injected into a website through XSS, it could access cookies containing sensitive data. Setting the HttpOnly flag prevents this.

  • If cookies are transmitted over unencrypted HTTP, they can be intercepted in man-in-the-middle attacks. The Secure flag ensures cookies are only sent over HTTPS.

  • Cookies automatically sent with requests can be exploited in CSRF attacks to perform unauthorized actions. Implementing CSRF tokens and using the SameSite attribute helps mitigate this.

  • Browsers impose size limits, typically 4KB per cookie, and quantity limits per domain. Too many cookies can lead to performance issues or request rejection.

Some best practices for secure cookie handling are as follows:

  • We should always set the HttpOnly flag to prevent JavaScript access and mitigate XSS risks. The Secure flag ensures cookies are sent only over HTTPS.

  • Using the SameSite attribute (SameSite=Strict or SameSite=Lax) restricts cross-site requests and reduces CSRF risks.

  • We should use short expiration times for sensitive data and avoid storing sensitive information like passwords or credit card numbers in cookies, storing only tokens or identifiers instead.

Sessions

Sessions are a server-side mechanism used to store and manage user authentication data during an active interaction with an application. Unlike cookies, which store data on the client side, sessions keep the data securely on the server, with the client holding only a reference, usually a session ID.

The authentication flow with sessions involves several steps:

  • During user login, the user submits credentials to log in. The server authenticates the credentials and creates a session for the user, typically represented by a unique session ID.

  • For session ID generation, the server creates a random, unique identifier linked to the user’s session data stored on the server. This session ID is sent to the client in a cookie or via another transport mechanism.

  • The server stores the session data (such as user ID, roles, and preferences) in memory, a database, or another storage system.

  • The session ID acts as a key to retrieve this data. For each subsequent request, the client sends the session ID, usually via a cookie. The server validates the session ID, retrieves the associated data, and uses it to process the request.

Sessions can have a time-to-live (TTL) to automatically expire after a period of inactivity or after a maximum duration. When the user logs out or a session is revoked due to security concerns, the session data is deleted from the server, and the session ID becomes invalid.

Relationship Between Sessions and Cookies

Cookies and sessions often work together.

Cookies hold the session ID on the client side. For each request, the browser sends the cookie to the server, allowing the server to identify the user’s session.

Here’s a simple code example showing how a session ID is set to the cookie:

app.post(’/login’, (req, res) => {
    const sessionId = createSession(req.body.username);
    res.cookie(’session_id’, sessionId, {
        httpOnly: true,
        secure: true,
        maxAge: 3600000
    });
    res.send(’Login successful’);
});

For incoming requests, the session ID stored in the cookie can be read by the server:

app.get(’/profile’, (req, res) => {
    const sessionId = req.cookies[’session_id’];
    if (isValidSession(sessionId)) {
        res.send(’Profile page’);
    } else {
        res.status(401).send(’Unauthorized’);
    }
});

Session Storage Options

Choosing the right session storage mechanism is critical for performance and scalability.

  • In-memory storage keeps sessions in the server’s RAM, providing the fastest performance. However, it’s not suitable for production environments because session data is lost when the server restarts. It also limits the application to a single server, making it unsuitable for distributed systems.

  • Database storage uses relational databases like PostgreSQL or MySQL, or NoSQL databases like MongoDB. This approach provides persistent and reliable session storage. However, we need to consider performance implications and ensure proper indexing on session ID columns for fast lookups. Database storage is reliable but may be slower than cache-based solutions.

  • Distributed cache systems like Redis or Memcached are high-performance key-value stores built specifically for session management. They offer excellent performance, built-in Time-To-Live (TTL) support for automatic session expiration, and the ability to scale horizontally. Redis is particularly popular for session storage in production environments due to its speed and reliability.

We should use in-memory storage only for development or testing. For production applications with a single server, database storage is adequate. For distributed systems or applications requiring high performance and scalability, Redis or Memcached is the recommended choice.

Distributed Session Strategies

When running multiple servers behind a load balancer, we face the challenge of ensuring all servers can access the same session data.

Sticky sessions (also called session affinity) is an approach where the load balancer routes all requests from a specific user to the same server. This is simple to implement because sessions don’t need to be shared across servers. However, it can lead to uneven load distribution, and if a server fails, users lose their sessions.

Session replication involves copying session data across all servers in the cluster. This provides high availability because if one server fails, others have the session data. However, it creates significant network overhead as sessions have to be continuously synchronized, and maintaining consistency across servers can be challenging in large clusters.

Centralized session store uses a single source of truth, typically a Redis cluster, where all servers read and write session data. This approach is highly scalable and provides consistency across all servers. The main concern is that the session store can become a single point of failure, but this can be mitigated by using Redis clustering or replication. This strategy is recommended for microservices architectures.

Challenges of Using Sessions

In distributed systems like microservices, session data must be shared across servers, requiring additional infrastructure like centralized databases or sticky sessions. These solutions add complexity and potential bottlenecks.

Also, if an attacker steals a session ID, they can potentially impersonate the user. This can be mitigated by using HTTPS to encrypt data in transit and implementing session timeouts.

Large numbers of active sessions can consume significant server resources, though this can be addressed by using efficient session storage like Redis.

Authentication with JWTs

JSON Web Tokens (JWT) are a compact, URL-safe, self-contained token format used to securely transmit information between parties.

JWTs are stateless, which means the server does not need to store session data. All user-related information is contained within the token. The server only needs the signing key to validate the token, making JWTs ideal for distributed systems where multiple services interact.

This is the reason JWTs are often employed for authentication and authorization to enable stateless and scalable systems.

Structure of a JWT

A JWT consists of three parts: header, payload, and signature. Each part is encoded in Base64.

The header contains metadata about the token, including the type (JWT) and the signing algorithm, like HS256 or RS256. Here’s an example:

{
    “alg”: “HS256”,
    “typ”: “JWT”
}

The payload contains the claims, which are statements about the user or token. Claims can be registered claims (predefined, such as iss for issuer, exp for expiration time, sub for subject), public claims (custom claims defined by the application), or private claims (custom claims shared between the issuer and consumer). Here’s an example:

{
    “sub”: “1234567890”,
    “name”: “John Doe”,
    “admin”: true,
    “iat”: 1516239022
}

Lastly, the signature ensures the integrity of the token and verifies the authenticity of the sender. It is created by encoding the header and payload and signing them with a secret or private key:

HMACSHA256(
    base64UrlEncode(header) + “.” + base64UrlEncode(payload),
    secret
)

Here’s how JWT works in a typical authentication flow:

  • After a user logs in, the server authenticates the credentials and generates a JWT containing claims about the user, such as the user ID and roles.

  • The JWT is signed using a secret key (symmetric signing) or a private key (asymmetric signing). The signed token is sent to the client, typically in the response body or as a cookie.

  • For subsequent requests, the client includes the JWT in the Authorization header as a Bearer token.

  • The server validates the token by verifying the signature and checking claims like expiration (exp) or audience (aud).

  • If the token is valid, the server allows access to the requested resource.

Refresh Token Flow

A common challenge with JWTs is that access tokens should be short-lived for security reasons, typically lasting 15 minutes to 1 hour. However, forcing users to log in repeatedly creates a poor user experience.

Refresh tokens solve this problem by enabling seamless reauthentication without requiring users to enter their credentials again. The pattern uses two types of tokens.

  • The access token is short-lived and used for authenticating API requests.

  • The refresh token is long-lived (days to weeks) and used only to obtain new access tokens. Refresh tokens are stored securely, typically in HttpOnly cookies or secure storage on mobile devices, to prevent theft.

The complete flow works as follows:

  • The user logs in with credentials, and the server validates them.

  • The server issues both an access token and a refresh token.

  • The client stores both tokens securely.

  • For subsequent requests, the client uses the access token in the Authorization header.

  • When the access token expires, the client sends a request to a refresh endpoint with the refresh token.

  • The server validates the refresh token and checks if it’s still valid and not revoked.

  • If valid, the server issues a new access token (and optionally a new refresh token).

  • The client uses the new access token for subsequent requests.

This approach balances security and user experience. Short-lived access tokens limit the damage if they’re stolen, while refresh tokens allow users to stay authenticated without constantly re-entering credentials.

PASETO

PASETO, short for Platform-Agnostic Security Tokens, is a modern alternative to JWTs designed with a focus on security, simplicity, and cryptographic best practices.

PASETO addresses some common vulnerabilities and misuse issues associated with JWT while maintaining the flexibility needed for a wide range of authentication and authorization scenarios.

Unlike JWT, which allows flexibility in algorithm choice (sometimes leading to insecure configurations), PASETO enforces the use of strong, cryptographically sound algorithms, reducing the risk of vulnerabilities from misconfiguration. While JWT allows developers to use weak algorithms like HS256 or the infamous none algorithm, PASETO restricts the choice to robust algorithms like AES-GCM for encryption (symmetric) and Ed25519 for signing (asymmetric).

PASETO tokens are either local (encrypted) or public (signed), ensuring that sensitive data is either securely encrypted or signed for authenticity. In contrast, JWT tokens are always visible in plaintext, even when signed, making them prone to unintentional information leaks.

Structure of a PASETO

A PASETO token consists of three or four main components, depending on whether it is a local token or a public token. These components are separated by dots.

  • The version indicates the version of the PASETO protocol being used. For example, version v1 uses older cryptographic standards, whereas version v2 uses modern, secure cryptographic standards.

  • The purpose specifies the type of the token, whether it is meant to be encrypted or signed. The options are local (encrypted token) and public (signed token).

  • The payload contains the actual data or claims within the token. For local tokens, the payload is encrypted and appears as an opaque Base64Url-encoded string. For public tokens, the payload is plaintext and Base64Url-encoded.

  • The footer contains optional metadata not included in the payload, such as audience (aud) or issuer (iss).

Local PASETO vs Public PASETO

Local PASETO tokens are encrypted, ensuring the confidentiality of the data contained within the token. They are intended for scenarios where sensitive information must remain hidden from unauthorized parties. They use symmetric encryption algorithms to ensure the payload is confidential. Only parties with the shared secret key can decrypt and access the token’s contents.

Public PASETO tokens are signed. They ensure the integrity of the data but not its confidentiality. These tokens are transparent and can be read by anyone, but only verified by the holder of the signing key. In other words, the token cannot be altered without invalidating the signature. They use asymmetric cryptography (for example, ED25519) to sign the token. The public key is used to verify the token, while the private key is used to sign it. They are suitable for scenarios where the payload needs to be read by clients but must remain tamper-proof.

Challenges of PASETO

Compared to the ubiquitous JWT, PASETO is relatively new and less widely adopted. This means fewer libraries, tools, and community support are available.

Developers familiar with JWT may need time to understand and adapt to PASETO’s principles and features. While the PASETO standard is well-defined, its ecosystem lacks the extensive middleware and framework integrations available for JWT.

Summary

In this article, we’ve looked at multiple authentication techniques in detail. Here are the key learning points in brief:

  • Authentication is the process of verifying user identity and is fundamental to application security. It is distinct from authorization, which determines what authenticated users can do.

  • Cookies store small pieces of data on the client side and use attributes like HttpOnly, Secure, and SameSite to control security and behavior.

  • First-party cookies are set by the domain we visit and are essential for authentication, while third-party cookies are set by external domains and are increasingly restricted by browsers.

  • Sessions store authentication data server-side and use cookies to hold only a session ID reference, providing better security than storing sensitive data client-side.

  • Session storage options include in-memory (development only), database storage (single server), and distributed caches like Redis (production and distributed systems).

  • Distributed session strategies include sticky sessions (simple but problematic), session replication (high overhead), and centralized session stores (recommended for microservices).

  • JWT is a self-contained, stateless token format with three parts (header, payload, signature) that eliminates the need for server-side session storage.

  • Refresh tokens solve the JWT expiration problem by using short-lived access tokens for requests and long-lived refresh tokens for obtaining new access tokens.

  • JWT tokens cannot be easily revoked once issued, requiring additional mechanisms like token blacklists or short expiration times to manage security.

  • PASETO improves on JWT by enforcing strong cryptographic algorithms and offering both encrypted (local) and signed (public) token types.

  • PASETO eliminates JWT’s algorithm confusion vulnerabilities by removing algorithm negotiation and using opinionated, secure defaults.

  • Choosing between cookies/sessions (traditional web apps), JWT (distributed systems and APIs), or PASETO (security-critical applications) depends on our specific scalability, security, and architectural requirements.
