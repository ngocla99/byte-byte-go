Must-Know Message Broker Patterns

Modern distributed systems rely on message brokers to enable communication between independent services.

However, using message brokers effectively requires understanding common architectural patterns that solve recurring challenges. This article introduces seven essential patterns that help developers build reliable, scalable, and maintainable systems using message brokers.

These patterns address three core categories of problems:

  • Ensuring data consistency across services

  • Managing workload efficiently

  • Gaining visibility into the messaging infrastructure.

Whether we’re building an e-commerce platform, a banking system, or any distributed application, these patterns provide proven solutions to common challenges.

In this article, we will look at each of these patterns in detail and understand the scenarios where they help the most.

Ensuring Data Consistency

The first category where message brokers can help is ensuring data consistency, and here are some key patterns related to it:

Transactional Outbox Pattern

When a service needs to save data to a database and publish a message to a broker, a critical problem emerges.

If we save the data successfully, but the network fails before sending the message, the system becomes inconsistent. This is the dual write problem, where we’re attempting to write to two separate systems without a guarantee that both will succeed.

For example, consider creating an order. We need to insert order details into the Orders table and notify other services by publishing an OrderCreated message. If the database write succeeds but the message never reaches the broker due to a network issue, other services never learn about the order. This means that the inventory system won’t reserve stock, and payment processing won’t begin. In other words, the order exists in limbo.

The Transactional Outbox pattern solves this by treating the message as data that gets saved in the same database transaction as the business data. Instead of sending messages directly to the broker, we insert them into a special table called the Outbox within the same transaction. This guarantees atomicity since both operations succeed or fail together.

See the diagram below:

The detailed implementation flow is as follows:

  • Begin a database transaction

  • Insert order data into the Orders table

  • Insert the event payload into the Outbox table

  • Commit the transaction

  • A relay process polls the Outbox table for new entries and publishes messages to the broker

  • After confirmation, the relay deletes the processed entry

  • A separate relay process then reads from the Outbox table and publishes messages to the broker.

This process can use simple polling at regular intervals or more sophisticated change data capture mechanisms to detect new messages instantly. Once the broker confirms receipt, the relay deletes the message from the Outbox.

The main advantage is consistency. Even if the message broker is completely down when we’re saving data, the messages are safely stored in the database. The relay will send them when the broker becomes available again. The trade-off is added complexity since we need to implement and maintain the relay component, and messages experience slight latency between the database commit and actual publication.

Saga Pattern

Distributed transactions across multiple databases pose a problem in microservices architectures. Each service owns its data, and there’s no global transaction coordinator.

The Saga pattern provides an alternative by breaking a business process into a series of local transactions, where each service performs its work and publishes an event to trigger the next step.

For example, consider an e-commerce order process:

  • The Order Service creates a pending order and publishes an OrderCreated event.

  • The Payment Service then picks up this event, processes the customer’s payment, and publishes a PaymentProcessed event.

  • The Shipping Service listens for this event and ships the item.

See the diagram below:

Each service executes its local transaction independently. If any step fails, compensating transactions undo previous work. For example, if payment processing fails due to insufficient funds, the Payment Service publishes a PaymentFailed event. The Order Service marks the order as cancelled. Each service reverses its local changes, effectively rolling back the distributed transaction.

See the diagram below:

This choreography approach is highly decoupled since services don’t directly call each other. They only know about events, not about other services. There’s no central coordinator that could become a single point of failure. Services can be developed, deployed, and scaled independently.

However, this pattern can create complex dependencies where understanding the complete flow requires examining multiple services. When a saga fails, we need to trace through event logs across services to understand what happened. Debugging becomes more challenging compared to orchestration-based approaches, where a central component manages the workflow and maintains a clear view of the entire process state.

Event Sourcing with CQRS

Traditional databases store only the current state of the data. If an account balance is $500, that’s all we see. The history of how it reached $500 is lost.

Event Sourcing takes a fundamentally different approach by storing every state change as an immutable event. Instead of updating a record in place, we append events to an event store. The current state can be reconstructed by replaying these events.

For example, rather than storing an account balance of $500, event sourcing would store the complete history:

  • AccountCreated with $0

  • MoneyDeposited for $1000

  • MoneyWithdrawn for $500.

Replaying these events in sequence produces the current $500 balance. This provides a complete audit trail and lets us rebuild the state at any point in time.

See the diagram below:

The event store acts as the source of truth. Events are immutable, meaning once written, they never change. For entities with thousands of events, replaying from the beginning becomes expensive, so we can create periodic snapshots. This way, replays can start from the most recent snapshot rather than event zero.

CQRS (Command Query Responsibility Segregation) pairs naturally with Event Sourcing by separating write operations from read operations. The write side validates commands, enforces business rules, and publishes events. The read side subscribes to these events via a message broker and maintains denormalized views optimized for specific queries.

See the diagram below:

For example, when a user transfers money, the write model validates that sufficient funds exist, then appends a MoneyTransferred event to the event store. This event flows through the message broker to multiple read models. One might update a balance view in a SQL database. Another might update transaction history in a document store. A third might feed analytics into a time-series database.

The main advantage is flexibility in querying data. We can create multiple specialized read models without affecting the write side. However, the challenge is eventual consistency since read models update asynchronously after events are published.

Efficient Workload Management

The next category where message brokers help is managing workload efficiently. Here are the key patterns for the same:

Competing Consumers Pattern

A single consumer processing messages from a queue often can’t keep pace with a fast producer. Messages accumulate in the queue, processing latency increases from seconds to minutes or hours, and the system falls behind. Upgrading hardware for a single consumer helps initially, but quickly hits diminishing returns.

The Competing Consumers pattern solves this by running multiple instances of the consumer service in parallel. Multiple consumers connect to the same queue and compete for messages. The message broker becomes a load balancer, distributing messages among available consumers.

See the diagram below:

If a producer sends four messages and two consumers are running, the broker might deliver messages one and three to the first consumer while sending messages two and four to the second consumer. Both consumers process their assigned messages simultaneously, effectively doubling throughput without any changes to the code.

This pattern also improves availability. If one consumer crashes due to a bug or hardware failure, other consumers continue processing messages. The broker detects the failure and redistributes pending work to the remaining healthy consumers. In other words, the system degrades gracefully rather than stopping completely.

The trade-off is loss of global message ordering. Since multiple consumers process messages concurrently, message two might complete before message one finishes. For many use cases, such as processing independent user registrations, the order doesn’t matter. However, for scenarios like processing bank transactions for a single account, order is critical.

If strict ordering is required, we need partitioning strategies like Kafka’s consumer groups, where messages with the same key always go to the same consumer instance. This adds complexity to the system but maintains ordering guarantees within each partition.

Priority Queue Pattern

Not all messages deserve equal treatment. In most systems, urgent customer requests should be processed before routine background tasks. For example, a fraud alert needs immediate attention, while a nightly report can wait. However, a single queue treats all messages equally, processing them strictly in arrival order. This means high-priority messages might end up waiting behind low-priority ones that happened to arrive earlier.

The Priority Queue pattern uses separate queues for each priority level with dedicated consumer pools. Instead of one queue with mixed priorities, we can create distinct queues: high priority, normal priority, and low priority. Consumers are allocated proportionally, with more consumers assigned to higher-priority queues.

For example, with ten total consumers, we might allocate four to the high-priority queue, three to normal priority, and three to low priority. This ensures urgent messages get processed quickly, even when the system is under heavy load. During traffic spikes, we can dynamically shift consumers from lower-priority queues to handle critical messages faster.

See the diagram below:

Priority assignment can happen in several ways.

  • The simplest approach is having producers explicitly set priority metadata when publishing messages. A fraud detection service marks its alerts as high priority. A reporting service marks its requests as low priority.

  • Alternatively, content-based routing examines message content and routes automatically. A router might check if the order total exceeds $10,000 or if the customer has VIP status, then route to the high-priority queue.

  • More sophisticated systems calculate priority scores based on multiple factors like customer tier, service level agreement deadlines, and order value.

The main challenge is starvation, where low-priority messages never get processed because high-priority messages keep arriving. Mitigation strategies for this situation include aging, where the message priority gradually increases the longer it waits. For example, a low-priority message waiting for two hours might be promoted to normal priority.

Claim Check Pattern

Message brokers often have size limits on individual messages, typically ranging from a few hundred kilobytes to a few megabytes. Sending large payloads like video files, high-resolution images, or massive datasets through the broker can hit these limits or degrade performance. Even if technically possible, routing gigabytes of data through the broker wastes bandwidth and slows down all message processing.

The Claim Check pattern solves this by separating the message from its payload. When a producer has large amounts of data to send, it first uploads the data to external storage like Amazon S3, a database, or a file server. Then it sends only a lightweight reference token (the claim check) through the broker. This token might be a storage URL, a database ID, or a file path.

The consumer receives the claim check from the broker, uses it to retrieve the actual data from storage, processes the data, and optionally deletes it from storage when finished. The broker only handles small reference messages, maintaining high throughput regardless of actual payload sizes.

This approach keeps your broker fast and avoids size limit errors. The trade-off is additional complexity since you need to manage external storage, handle retrieval failures, and decide on cleanup policies for stored data.

Gaining System Visibility

Another category of requirements where message brokers help is gaining system visibility. Here are the patterns for the same:

Scatter-Gather Pattern

Sometimes we need information from multiple sources to answer a single request. For example, a user asks for the best price on a product, but that data lives in different inventory systems. Or they want flight options, but each airline runs an independent booking system.

The Scatter-Gather pattern broadcasts a request to multiple recipients and combines their responses into one unified answer.

For reference, consider a flight booking system. When a user searches for flights from New York to Los Angeles, we need prices from multiple airlines. The scatter component publishes the search request to a common topic. Each airline service that subscribes to this topic receives the request, queries its internal systems, and publishes results back to a response topic.

An aggregator component collects these responses using a correlation ID that links responses to the original request. The aggregator knows it expects responses from three airlines, so it waits for all three or until a timeout expires. After collecting responses, it combines them according to business logic, like sorting by price, filtering by departure time, or selecting the single cheapest option.

The pattern supports two distribution strategies.

  • The recipient list approach maintains explicit knowledge of each service to query. For example, the scatter component knows to send requests to Service A, Service B, and Service C queues. This provides control but creates coupling. Adding a new service means updating the scatter component configuration.

  • The publish-subscribe approach broadcasts to any interested service. The scatter component publishes to a generic topic. Any service can subscribe without the scatter component knowing about it. New services join by simply subscribing to the topic. This removes coupling but relinquishes control over who responds.

The aggregator must handle partial failures gracefully. Most implementations use timeouts to prevent one slow service from degrading overall response time.

Wire Tap Pattern

Production debugging and monitoring require visibility into message flows without disrupting normal operations.

The Wire Tap pattern copies messages flowing through a channel to a separate monitoring channel while allowing the original message to continue unaffected to its destination. For example, imagine messages flowing from a producer through a queue to a consumer. A wire tap component sits transparently between them, forwarding each message to the consumer as normal but also sending a copy to a monitoring queue. This monitoring queue feeds specialized services for logging, analytics, audit trails, or debugging.

The main characteristic is that the tap operates in fire-and-forget mode. It doesn’t wait for the monitoring consumer to acknowledge receipt of the copy. If the monitoring system is slow or down completely, that should never impact the primary business flow. The original message continues to its destination regardless of what happens to the copy.

Common use cases include compliance logging, where regulations require recording every financial transaction. For example, banking systems may tap payment processing flows to create immutable audit trails. E-commerce companies tap order flows to feed analytics systems analyzing purchasing patterns.

Summary

In this article, we have looked at multiple patterns that work with message brokers as the center. Here are the key learning points in brief:

  • The Transactional Outbox pattern solves the dual write problem by saving messages in the same database transaction as the business data, guaranteeing that messages are never lost even if the broker is temporarily unavailable.

  • Saga patterns enable distributed transactions across microservices through choreographed events and compensating transactions that undo work when steps fail, eliminating the need for global transaction coordinators.

  • Event Sourcing stores the complete history of state changes as immutable events rather than just the current state, providing a full audit trail and the ability to rebuild state at any point in time.

  • CQRS separates write operations from read operations, allowing us to maintain multiple specialized read models optimized for different queries while the write side focuses on business logic validation.

  • Competing Consumers enables horizontal scaling by running multiple consumer instances that share work from a single queue, making it trivial to increase processing capacity by launching additional instances.

  • Priority Queue patterns use separate queues for different priority levels with proportional consumer allocation, ensuring urgent messages get processed immediately, even during high system load.

  • The Claim Check pattern allows large amounts of data to be transferred by separating the message from its payload

  • Scatter-Gather broadcasts requests to multiple services in parallel and aggregates their responses where we need data from independent sources.

  • The Wire Tap pattern copies messages to monitoring channels in fire-and-forget mode, providing visibility into message flows for debugging and analytics without impacting primary business operations.
