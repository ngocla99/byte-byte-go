A Guide to Relational Database Design

The hardest part of relational database design is not using SQL. The syntax for creating tables, defining keys, and writing joins can be learnt and mastered over time. The difficult part is to develop the thinking that comes before any code gets written, and answering questions about the design of the database.

  • Which pieces of information deserve their own table?

  • How should tables reference each other?

  • How much redundancy is too much?

These are design decisions, and getting them right means our data stays consistent, our queries stay fast, and changes are painless. Getting them wrong means spending months patching problems that were baked into the structure from day one.

In this article, we cover the core concepts that inform those decisions. We’ll look at tables, keys, relationships, normalization, and joins, with each concept building on the last.

Tables and the Language That Drives Them

A relational database organizes data into tables, also called relations. Each table represents a specific entity, so a Customers table holds customer data, an Orders table holds order data, and a Products table holds product data.

Every table is made up of rows and columns. A row (sometimes called a record) represents a single instance of that entity, one specific customer, one specific order. A column (also called a field or attribute) represents a property of that entity, like a customer’s name or an order’s date.

To interact with these tables, we use SQL (Structured Query Language). SQL is the standard language for relational databases, and its syntax is largely the same whether we’re working with MySQL, PostgreSQL, or Oracle. SQL supports four fundamental operations, commonly known as CRUD.

  • Create inserts new records.

  • Read retrieves data based on criteria.

  • Update modifies existing records.

  • Delete removes records.

These four operations, combined with commands for defining table structure, provide everything needed to build and manage a database.

Deciding what gets its own table versus what stays as a column on an existing table is one of the most important decisions in database design. That decision becomes clearer once we understand how to identify records and connect tables, which brings us to keys.

Keys: Identity and Connection

Keys solve two main problems in database design:

  • First, they give each record a unique identity.

  • Second, they create the links between tables.

Without keys, we’d have a collection of isolated spreadsheets with no way to connect them reliably.

Primary Keys and Indexing

A primary key is a column (or combination of columns) that uniquely identifies each row in a table. No two rows can share the same primary key value, and the value can never be null.

CREATE TABLE books (
    book_id INT PRIMARY KEY,
    title VARCHAR(100),
    author VARCHAR(100),
    isbn VARCHAR(20) UNIQUE,
    publication_year INT,
    genre VARCHAR(50)
);

Here, book_id is the primary key. Every book gets a unique ID, and we can always find a specific book by referencing that ID.

When we define a primary key, the database automatically creates an index on that column. An index is a separate data structure, a sorted reference that lets the database locate specific records without scanning every row in the table. Think of it like the index at the back of a textbook, where instead of flipping through every page, we look up the term and jump straight to the right page.

Surrogate Keys versus. Natural Keys

Not all primary keys are created equal. A surrogate key is an artificial identifier generated by the database, often an auto-incrementing integer. It has no inherent meaning.

CREATE TABLE customers (
    customer_id INT PRIMARY KEY AUTO_INCREMENT,
    first_name VARCHAR(50),
    last_name VARCHAR(50),
    email VARCHAR(100) UNIQUE
);

A natural key, by contrast, comes from the data itself, like a product code, an ISBN, or an email address that already uniquely identifies the record based on real-world attributes.

CREATE TABLE products (
    product_code VARCHAR(20) PRIMARY KEY,
    product_name VARCHAR(100),
    category VARCHAR(50),
    price DECIMAL(10, 2)
);

The tradeoff is real. Surrogate keys are stable and simple. They never change, and they make relationships between tables easy to manage. Natural keys are meaningful and sometimes eliminate the need for extra lookups, but they can change (people switch email addresses), and composite natural keys add complexity. Most teams default to surrogate keys, but it’s worth understanding both options rather than blindly following convention.

Foreign Keys and Referential Integrity

A foreign key is a column in one table that references the primary key of another table. This is the mechanism that connects tables with each other.

CREATE TABLE customers (
    customer_id INT PRIMARY KEY,
    first_name VARCHAR(50),
    last_name VARCHAR(50),
    email VARCHAR(100)
);

CREATE TABLE orders (
    order_id INT PRIMARY KEY,
    customer_id INT,
    order_date DATE,
    total_amount DECIMAL(10, 2),
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
);

But the foreign key’s real purpose goes beyond linking. It enforces referential integrity, which means the database guarantees that every foreign key value actually points to an existing record in the parent table. Without this constraint, we could end up with orders pointing to customers who don’t exist, creating orphaned records that corrupt the data and break queries.

When a referenced record in the parent table is updated or deleted, the constraint defines what happens next.

  • CASCADE automatically propagates the change to the child table (delete a customer, and their orders get deleted too).

  • SET NULL sets the foreign key value to NULL in the child table.

  • NO ACTION prevents the deletion entirely if child records exist.

CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    CustomerID INT,
    OrderDate DATE,
    FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID) ON DELETE CASCADE
);

Foreign key constraints aren’t optional syntax decoration. Without them, application code becomes solely responsible for preventing data corruption. The database is a much safer place to enforce these rules.

Now that we can identify records and link tables together, we need to understand the patterns those links follow.

Relationships: How Entities Connect

When we look at business requirements, one of the first design tasks is recognizing how entities relate to each other. These relationships fall into three patterns, and getting the pattern right determines whether the schema can accurately represent the data.

One-to-One

Each record in Table A maps to exactly one record in Table B, and vice versa. For example, a user has one profile. A profile belongs to one user.

CREATE TABLE Users (
    UserID INT PRIMARY KEY,
    Username VARCHAR(50),
    Email VARCHAR(100)
);

CREATE TABLE UserProfiles (
    ProfileID INT PRIMARY KEY,
    UserID INT UNIQUE,
    FirstName VARCHAR(50),
    LastName VARCHAR(50),
    Bio VARCHAR(500),
    FOREIGN KEY (UserID) REFERENCES Users(UserID)
);

One-to-one relationships are relatively rare, and they often signal that the two tables could be merged into one. But they’re useful when we want to separate data for security or performance reasons. Maybe profile details are large and rarely accessed, or maybe sensitive information belongs in a table with tighter access controls.

One-to-Many

This is the most common relationship in relational databases. One customer can place many orders, but each order belongs to exactly one customer.

CREATE TABLE Customers (
    CustomerID INT PRIMARY KEY,
    FirstName VARCHAR(50),
    LastName VARCHAR(50),
    Email VARCHAR(100)
);

CREATE TABLE Orders (
    OrderID INT PRIMARY KEY,
    CustomerID INT,
    OrderDate DATE,
    TotalAmount DECIMAL(10, 2),
    FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID)
);

See the diagram below:

The implementation is clean. A foreign key on the “many” side (Orders) points back to the primary key on the “one” side (Customers). One column, and the relationship is established.

Many-to-Many

Sometimes, records on both sides can relate to multiple records on the other side. A product can belong to multiple categories, and a category can contain multiple products. Students enroll in multiple courses, and courses have multiple students.

This can’t be modeled with a single foreign key. We need a junction table (also called a bridge table or associative table) that sits between the two tables and holds pairs of foreign keys.

CREATE TABLE Products (
    ProductID INT PRIMARY KEY,
    ProductName VARCHAR(100),
    Description VARCHAR(500),
    Price DECIMAL(10, 2)
);

CREATE TABLE Categories (
    CategoryID INT PRIMARY KEY,
    CategoryName VARCHAR(50),
    Description VARCHAR(500)
);

CREATE TABLE ProductCategories (
    ProductCategoryID INT PRIMARY KEY,
    ProductID INT,
    CategoryID INT,
    FOREIGN KEY (ProductID) REFERENCES Products(ProductID),
    FOREIGN KEY (CategoryID) REFERENCES Categories(CategoryID)
);

The junction table pattern is non-obvious to beginners, but it comes up constantly in real-world design. Anytime the thought process is “this entity can have multiple of those, and those can have multiple of these,” that’s a many-to-many relationship, and it will need a third table to model it properly.

Relationships give the database structure, but structure alone doesn’t prevent redundancy. A table can be full of duplicated data even with proper keys and relationships. That’s the problem normalization solves.

Normalization: Eliminating Redundancy

Database normalization is the process of organizing tables to reduce redundancy and prevent data anomalies. Before we look at the normal forms themselves, it helps to understand the specific pain they prevent.

  • Insertion anomalies occur when we can’t add data without the presence of other, unrelated data. For example, if customer details and product details live in the same table, it might not be possible to add a new product without associating it with a customer.

  • Update anomalies arise when changing one fact means updating it in dozens or hundreds of rows. For example, if a customer’s address is stored on every order row, changing that address means touching every single order.

  • Deletion anomalies happen when deleting one record accidentally destroys unrelated information. For example, if student and course data live in the same table, deleting the last student in a course might delete the course itself.

These are the reasons normalization exists. The normal forms are just the labels for each level of fix.

First Normal Form (1NF)

The rule is simple. Every cell in a table must hold a single, atomic value. There should be no comma-separated lists, no arrays stuffed into a column.

For example, the Courses column violates this by holding multiple values in a single cell.

CREATE TABLE Students (
    StudentID INT PRIMARY KEY,
    Name VARCHAR(100),
    Courses VARCHAR(200)
);

INSERT INTO Students (StudentID, Name, Courses)
VALUES (1, ‘John Doe’, ‘Math, Science, History’);

The fix is to break this out into separate tables with a proper many-to-many relationship.

CREATE TABLE Courses (
    CourseID INT PRIMARY KEY,
    CourseName VARCHAR(100)
);

CREATE TABLE StudentCourses (
    StudentID INT,
    CourseID INT,
    PRIMARY KEY (StudentID, CourseID),
    FOREIGN KEY (StudentID) REFERENCES Students(StudentID),
    FOREIGN KEY (CourseID) REFERENCES Courses(CourseID)
);

Second Normal Form (2NF)

2NF targets partial dependencies. Every non-key column must depend on the entire primary key, not just part of it. This only matters when the table has a composite primary key (a primary key made up of multiple columns).

Consider this table, where the primary key is the combination of OrderID and ProductID.

CREATE TABLE Orders (
    OrderID INT,
    ProductID INT,
    CustomerID INT,
    CustomerName VARCHAR(100),
    OrderDate DATE,
    PRIMARY KEY (OrderID, ProductID)
);

The problem is that CustomerName depends on CustomerID alone, not on the full composite key. It has no relationship with ProductID. The fix is to move customer data into its own table.

CREATE TABLE Customers (
    CustomerID INT PRIMARY KEY,
    CustomerName VARCHAR(100)
);

CREATE TABLE Orders (
    OrderID INT,
    ProductID INT,
    CustomerID INT,
    OrderDate DATE,
    PRIMARY KEY (OrderID, ProductID),
    FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID)
);

Third Normal Form (3NF)

3NF goes one step further and targets transitive dependencies. A transitive dependency exists when a non-key column depends on another non-key column rather than directly on the primary key.

CREATE TABLE Employee (
    EmployeeID INT PRIMARY KEY,
    Name VARCHAR(100),
    DepartmentID INT,
    DepartmentName VARCHAR(100),
    DepartmentLocation VARCHAR(100)
);

Here, DepartmentName and DepartmentLocation depend on DepartmentID, not on EmployeeID. The department information is just along for the ride. The fix is to extract the department data into its own table.

CREATE TABLE Department (
    DepartmentID INT PRIMARY KEY,
    DepartmentName VARCHAR(100),
    DepartmentLocation VARCHAR(100)
);

CREATE TABLE Employee (
    EmployeeID INT PRIMARY KEY,
    Name VARCHAR(100),
    DepartmentID INT,
    FOREIGN KEY (DepartmentID) REFERENCES Department(DepartmentID)
);

Most developers stop at 3NF. Higher normal forms exist (BCNF, 4NF, 5NF), but they address edge cases that rarely cause problems in practice. It’s also worth knowing that in read-heavy systems, teams sometimes deliberately denormalize, accepting some redundancy to avoid complex joins and speed up queries. The rules aren’t sacred, and knowing when to bend them is part of the decision-making process.

Normalization solves redundancy by splitting data across multiple tables. But now we need a way to put that data back together when querying it. That’s where joins come in.

Joins: Querying Across Tables

Joins combine rows from two or more tables based on a related column between them. If normalization is the process of taking data apart, joins are the process of putting it back together.

Inner Join

An inner join returns only the rows where there is a match in both tables. If a customer has placed no orders, they won’t appear in the results.

SELECT Customers.CustomerName, Orders.OrderDate, Orders.TotalAmount
FROM Customers
INNER JOIN Orders ON Customers.CustomerID = Orders.CustomerID;

This gives us every customer-order pair where the customer ID matches. Customers without orders and orders without valid customers are both excluded.

Left Join

A left join returns all rows from the left table, with matched data from the right table where available. If there’s no match, the right side comes back as NULL.

SELECT Customers.CustomerName, Orders.OrderDate, Orders.TotalAmount
FROM Customers
LEFT JOIN Orders ON Customers.CustomerID = Orders.CustomerID;

This is particularly useful for finding gaps, like customers who haven’t placed any orders yet. Every customer appears in the results, whether they have orders or not.

Right Join

A right join is the mirror image. It returns all rows from the right table, with matches from the left where available.

SELECT Departments.DepartmentName, Employees.EmployeeName
FROM Employees
RIGHT JOIN Departments ON Employees.DepartmentID = Departments.DepartmentID;

This is useful for finding departments with no employees, for example. Every department appears in the results, even if nobody works there yet.

Joins feel complex at first, but they’re the payoff of good design. If the tables are well-structured with proper keys and relationships, joins are simple and fast. If we avoided splitting tables to dodge joins, we’re probably living with the redundancy problems that normalization was meant to solve.

Summary

In this article, we’ve looked at relational database design in detail. Here are the key learning points in brief:

  • A relational database organizes data into tables (relations), where each table represents a specific entity, rows represent individual records, and columns represent attributes of those records.

  • SQL is the standard language for interacting with relational databases, supporting four core operations known as CRUD: Create, Read, Update, and Delete.

  • Primary keys uniquely identify each row in a table and cannot contain duplicate or null values. The database automatically creates an index on the primary key to speed up lookups.

  • Surrogate keys are artificial identifiers generated by the database, while natural keys are derived from the data itself. Most teams default to surrogate keys for their stability and simplicity, but natural keys are a valid choice when the data provides a reliable, unique identifier.

  • Foreign keys link tables together and enforce referential integrity, ensuring that every reference points to an existing record and preventing orphaned data. Constraint actions like CASCADE, SET NULL, and NO ACTION define what happens when a referenced record changes.

  • Relationships between tables follow three patterns: one-to-one (rare, useful for separating data), one-to-many (the most common, implemented with a foreign key), and many-to-many (requires a junction table to hold pairs of foreign keys).

  • Database normalization reduces redundancy and prevents data anomalies (insertion, update, and deletion) by progressively organizing data through normal forms. First Normal Form requires atomic values, Second Normal Form eliminates partial dependencies, and Third Normal Form eliminates transitive dependencies.

  • Joins are the mechanism for querying data across normalized tables. Inner joins return only matched rows, left joins include all rows from the left table regardless of matches, and right joins do the same for the right table.
