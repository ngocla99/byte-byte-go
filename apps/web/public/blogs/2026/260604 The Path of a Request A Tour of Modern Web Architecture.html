The Path of a Request: A Tour of Modern Web Architecture

A web page loads in under a second. In that second, a single user request may have passed through roughly ten distinct systems on its way to and from the database. The page feels fast because of how those systems are arranged. Each layer absorbs as much traffic as it can before passing the rest along. Taken together, the layers form a funnel, with most traffic handled long before it reaches the bottom.

Understanding what each layer does to narrow that funnel can lead to a better grasp of each component of a modern web stack.

In this article, we follow the journey of a web request one hop at a time. At each stop, we ask two questions. What is this layer doing, and what trade-off is it making? The journey starts before the request has fully left the browser, and latency is spent at every hop.

DNS

Before a request goes anywhere, the browser needs to know where to send it. The URL contains a name like example[dot]com, but networks route by numerical addresses. DNS is the protocol that translates between the two.

The mechanism is hierarchical.

The browser asks a recursive resolver, usually run by the network provider or by a public service. The resolver, in turn, asks the root servers, then the servers responsible for the top-level domain such as .com or .io, and finally the servers authoritative for the specific domain. The answer is an IP address. Every level caches results, which means most lookups never traverse the full chain.

The trade-off is straightforward. The first lookup costs tens of milliseconds, sometimes more across long distances, while subsequent lookups are effectively free thanks to caching. What this buys is decoupling. Servers can be replaced, scaled out, or moved across continents without users noticing anything, because the name they use stays the same.

DNS is also one of the first places to investigate when a system appears unreachable, but the underlying servers are healthy. A significant share of incidents that present as application failures turn out to be DNS issues at the resolver level.

Once the browser has an address, the next stop is rarely the application server itself. It is a server positioned closer to the user that can answer on the application’s behalf.

CDN

A content delivery network is a fleet of servers distributed across data centers around the world, each holding a cached copy of the application’s responses.

When a request arrives, it is routed to the nearest point of presence, often through DNS-based routing combined with anycast. For reference, anycast is a technique where the same IP address is advertised from multiple locations so that traffic naturally reaches the closest one. If the requested content is cached there, the response comes back from a few hundred kilometers away rather than from the origin server, which may be on a different continent.

The reason this matters comes down to physics. A round trip between distant continents takes approximately 80 milliseconds even under ideal network conditions, and each additional network layer adds further latency. Placing a copy of the response near the user avoids the round trip entirely.

The trade-off is a few milliseconds of edge processing in exchange for two substantial benefits. One is significant latency reduction for cacheable content, and the other is origin offload.

A well-tuned CDN handles the majority of static traffic without ever forwarding it deeper. For sites with high traffic, the application servers see only a fraction of what enters at the edge.

Modern CDNs increasingly run application code in addition to serving files. Edge functions execute logic at the point of presence, which is starting to blur the boundary between the network and the application.

When the CDN cannot serve the request from its cache, or when the request itself is dynamic and uncacheable, the journey continues to the origin’s edge. The first layer it meets is the system that distributes traffic among the machines running the application.

Load Balancer

At the front of any system running on more than one machine sits a load balancer. Its job is to receive every incoming request and forward it to one of many backend servers. The choice of which server gets which request is made by an algorithm, and the algorithm matters more than it first appears.

The simplest strategy is round robin, where each incoming request goes to the next server in the rotation. This is the easiest algorithm to explain, but real systems rarely use it alone. Some backends are slower than others, some are handling longer requests, and an even split of requests does not guarantee an even split of work. Production systems use strategies such as least connections, weighted random, or consistent hashing, each of which trades one form of fairness for another.

Load balancers also vary along a second dimension:

  • Layer 4 load balancers route by IP and port without inspecting request contents, which makes them fast.

  • Layer 7 load balancers read the HTTP request and can route by URL, header, or cookie, which makes them more capable but slower.

Many production systems run both, with layer 4 in front and layer 7 behind.

The trade-off is under a millisecond of decision time in exchange for horizontal scale, which removes the ceiling imposed by any single machine, and failure isolation, where a dead instance silently disappears from the pool while traffic continues flowing to the rest.

The load balancer has chosen a backend to handle the request. Before the request reaches the application code, it enters one more layer whose routing decisions are based on what the request is asking for rather than which machine should handle it.

API Gateway

An API gateway sits between external clients and the internal services that handle their requests. It is the single front door into a system that may have dozens or hundreds of services running behind it. When a request arrives, the gateway reads the URL or headers, decides which internal service should handle it, and forwards it accordingly.

A clean way to distinguish a gateway from a load balancer is to consider what each one routes on. A load balancer distributes traffic across identical backends that all do the same work, multiplying capacity by spreading load across many machines. On the other hand, a gateway routes among different services that each perform different functions, directing every request to whichever service handles that specific kind of work.

Gateways do more than route requests. They are also the place where cross-cutting concerns live. Rate limiting, authentication checks, request logging, format transformation, and basic security policy all tend to accumulate here. Implementing these in every service separately leads to drift and duplication, while handling them once at the gateway provides a single point of enforcement.

This concentration creates a tension. Gateways can accumulate responsibility until they become difficult to change without coordinating across many teams. The trade-off is a few milliseconds of routing logic in exchange for a stable client contract and a single place to enforce policy.

Before the gateway forwards the request along, there is one final question that needs an answer. The identity of the sender must be established before any internal service can rely on it.

Authentication

The system can only trust a request once it knows who is sending it. In most modern architectures, identity is carried in a token. The client presents the token in a request header, and some layer along the path, often the gateway and sometimes the service itself, checks it.

Tokens come in two main flavors:

  • An opaque token is a random string that points to a session record stored elsewhere, so verifying it requires looking up the session, which adds a few milliseconds.

  • A signed token, most commonly a JWT, carries its own claims inside it along with a cryptographic signature, so verifying it requires only a signature check, which takes well under a millisecond.

The convenience of JWTs comes with a real cost. Since they are valid for the duration encoded inside them, revoking access before expiry is difficult. They also grow with the data they carry, and any sensitive information inside one is exposed to anyone who can read it.

Authentication is distinct from authorization, with the former asking who the caller is and the latter asking whether the caller is allowed to do this specific thing. The two are different problems and are often handled by different systems.

The trade-off is under a millisecond for JWT verification and a few more for session lookups, in exchange for a trust boundary every downstream service can rely on. Once a request crosses that boundary, every subsequent service can assume identity has been established without needing to check it again.

With identity established, the request enters the internal network. In small systems, this is a direct call between services. In larger systems, every internal hop passes through one more piece of infrastructure first.

Service Mesh

In a system with many services, every service eventually needs to talk to others. The naive approach is for each service to make HTTP calls directly, which works at a small scale but fails to handle the cross-cutting concerns that inter-service communication actually requires. Retries, timeouts, encryption, observability, and traffic shifting need to be handled consistently across services written in many different languages.

A service mesh addresses this by attaching a small proxy, called a sidecar, to every service instance. All inter-service traffic flows through these proxies rather than directly between applications, and the proxies handle the cross-cutting concerns uniformly regardless of the language each service is written in.

The trade-off is one to three milliseconds per internal hop plus significant operational complexity, in exchange for consistent behavior across the system without requiring any application code changes. Whether this trade is worth making depends heavily on the size of the system, and many smaller and mid-sized systems run perfectly well without a mesh.

After all the layers are engineered to route, distribute, verify, and transport, the request finally arrives at the place where the actual work happens.

Service or Application

The service is where the application’s business logic finally runs. The request is received, the relevant code executes, often calling other services or talking to a cache and database in the process, and a response is assembled to send back.

This is the only layer in the path whose latency is quite variable. The infrastructure layers usually add small, predictable amounts of time, but the actual work depends entirely on what the request is asking for. A simple lookup might take a millisecond, while an aggregation across several data sources might take hundreds.

The defining feature of a service is independent deployability and clear ownership. There is no requirement that the code itself be small, and a correctly-sized microservice might run to tens of thousands of lines. What matters is the organizational boundary around the service, not the size of the codebase inside it.

The early enthusiasm for splitting systems into many fine-grained services has been moderated in more recent years, and a number of organizations have consolidated services they had previously separated. The right granularity depends on the size and structure of the engineering team, and the appropriate choice varies significantly across contexts.

The trade-off at this layer is whatever the work actually takes, in exchange for independent deployability, team autonomy, and the ability to use different technologies for different problems.

Cache

A cache is a fast key-value store that holds frequently accessed data, usually in memory. The most common pattern is called cache-aside. Before querying the database, the service checks the cache. On a hit, the answer comes back in well under a millisecond. On a miss, the service queries the database, stores the result in the cache, and returns it.

The latency difference between a cache and a database is substantial enough to shape application design. An in-memory cache responds in sub-millisecond time, while a database query, even an indexed one, takes several milliseconds at best. For data that is read often and changes rarely, caching turns a slow operation into an almost free one.

Caches also protect the database from a load it could not otherwise handle. A popular page on a busy site might be requested thousands of times per second, and hitting the database for each one would overwhelm it. Serving most of those requests from cache means the database sees only a fraction of the actual traffic.

The price of caching is invalidation. When the underlying data changes, the cached copy is now wrong, and deciding when and how to refresh it is one of the hard problems in distributed systems. Strategies range from short time-to-live values that accept some staleness to event-driven invalidation that aims for consistency at the cost of complexity.

A more subtle issue is the cache stampede. When a hot cache entry expires, many requests may notice simultaneously, and all hit the database at once, causing issues.

Database

The database is the ultimate system of record. It is where the authoritative version of the application’s data lives, and it is the only layer in the path that can give a definitive answer to any question about that data.

The mechanism of a database is more complex than it first appears:

  • A query is parsed, planned, and executed.

  • The planner decides which indexes to use, in what order to read from them, and how to combine results.

  • The executor reads from storage, which may be disk or memory, depending on what is cached internally.

Modern databases are themselves distributed systems with their own caches, replication, and storage engines, but from the outside, they appear as something that accepts queries and returns results.

The trade-off is in the range of milliseconds for an indexed read, much more for complex joins or table scans, in exchange for durability, consistency, and a rich query language. The cost reframe is worth carrying. A database is slower than a cache because it is doing more, and durability, consistency, and rich query planning all take work. That work is the price of being authoritative.

Most production databases run as a primary with several read replicas, where writes go to the primary and reads are spread across replicas. This arrangement trades a small amount of staleness for much higher read capacity. The broader spectrum of consistency and availability tradeoffs has shaped database design for decades.

Response Path

The response travels back through the same set of layers it came in through, but the work each layer does on the way back is different. Some examples:

  • The microservice serializes its result into a response body, while the service mesh proxy logs and observes the outgoing data.

  • The gateway may transform headers, set cookies, and apply final policies

  • The CDN may compress the response and, importantly, cache it for the next user who asks for the same thing.

This last behavior matters because the next request for the same content may never reach any of the inner layers at all, and it will be served entirely from the CDN. The funnel widens at the top over time as content accumulates in edge caches. The first user pays the cost of warming the cache, and every user who follows benefits from it.

The response trip is usually slightly faster than the outbound trip.

Summary

In this article, we’ve looked at each component on the request path in detail. Here are the key learning points in brief:

  • A web request passes through roughly ten distinct systems on its way to the database and back, and most requests never reach the bottom of that funnel.

  • Each layer makes a trade, paying a specific latency cost in exchange for a specific capability.

  • DNS, the CDN, and the load balancer between them absorb most traffic before it ever touches application code.

  • Load balancers enable horizontal scale, and the choice of algorithm matters far more than a simple round-robin example would suggest.

  • Gateways and load balancers are different kinds of routers, with gateways selecting which service should handle each request and load balancers distributing requests across identical backends.

  • Authentication is the construction of a trust boundary, and once a request crosses it, every downstream service can rely on identity having been established.

  • A service mesh handles inter-service communication uniformly, but plenty of correct systems are built without one.

  • Caches are as much a database protection mechanism as a speed optimization, and invalidation is one of the genuinely hard problems in distributed systems.

  • Databases are slower than caches because they do more, and durability, consistency, and rich query planning all take meaningful work.

  • Latency adds linearly across hops, but reliability multiplies. Ten hops at 99.9% reliability each compose to roughly 99%, which is why teams instinctively try to remove layers wherever they can.
