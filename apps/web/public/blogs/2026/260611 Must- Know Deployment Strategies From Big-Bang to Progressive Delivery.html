Must- Know Deployment Strategies: From Big-Bang to Progressive Delivery

Deployment is the moment when code stops being a developer’s problem and becomes everyone’s. It is the act of taking something that worked on a build server and putting it in front of real users, on real infrastructure, and handling real traffic. For a long time, this moment was riskier than it had any reason to be, and the strategies we will discuss in this article are what teams built to take that risk down.

Several distinct strategies are in common use today, and each one is an answer to a specific problem that the previous approaches couldn’t solve well enough. Some reduce the blast radius (the number of users affected) when a deploy goes wrong. Others separate the moment the code reaches production from the moment users actually see it.

In this article, we’ll go through the main deployment strategies used in production today, looking at how each one works, what it costs, and when it makes sense to use.

Big-Bang Deployment

Big-bang deployment is the simplest possible answer. Stop the old version of the service, start the new one, and accept that there will be a brief window when nothing is running. Users requesting the service during that window get an error or a maintenance page. Once the new version is up, everyone is able to access the application.

Big-bang is sometimes treated as outdated, but it remains the right strategy in many situations where simpler infrastructure is the rational choice. For example, a small internal tool used by twelve people during business hours can deploy at three in the morning with a maintenance message. A side project with five users can take a sixty-second downtime to ship a new version. The complexity of the strategies further down this list isn’t free, and a team that doesn’t need that complexity is making a rational choice when they skip it.

What big-bang gives up is graceful failure. If the new version is broken, every user encounters the breakage at once, and rolling back means doing another big-bang deploy of the old version, which itself can fail.

A more subtle failure mode happens when the deploy is supposed to update every server at the same time, but does not. Some servers received the update, others didn’t, and the system is now running two versions of itself in an uncontrolled way. This state is worse than either version alone, because nothing in the codebase was designed to handle the mixed condition.

Big Bang’s defining limit is that there has to be a moment when nothing is running.

Rolling Deployment

Rolling deployment takes the same job and spreads it out. Instead of stopping everything and starting everything, instances are updated a few at a time. While the rollout is in progress, traffic continues to flow, distributed across whatever mix of old and new instances happens to be running at that moment. By the time the last few instances flip, the whole fleet is on the new version.

This solves the downtime problem cleanly. There is no maintenance window, no error page, and no scheduled deploy at three in the morning. The system stays up throughout. For services that need to be continuously available, this alone makes rolling deployment worth the additional infrastructure, which includes a load balancer, multiple instances, and the ability to gracefully stop sending requests to instances that are about to be updated.

What rolling gives back is harder to see. During the rollout, the system is in a mixed state, with some users hitting the old version and some hitting the new. Any shared state, including cached objects, in-memory queues, or items in a database written by one version and read by another, has to make sense to both versions. If the new version writes a field that the old version doesn’t understand, weird behavior can be encountered.

The more dangerous problem is what we might call slow poisoning. Rolling deployment protects against immediate failures, such as the ones that crash the new version on startup. But a deploy with a subtle bug, say a memory leak or a degraded code path that only triggers for some users, won’t fail loudly during the rollout. It will complete, look successful, and then every instance in the fleet will start misbehaving at roughly the same time, an hour or a day later.

Blue-Green Deployment

Blue-green deployment trades infrastructure for safety.

The idea is to keep two identical production environments running at the same time, one of them serving real traffic and the other sitting idle. New versions get deployed to the idle environment, tested there as much as possible, and then a switch at the routing layer cuts all traffic over to the new environment at once. The previous environment stays running, untouched, ready to receive traffic again if something goes wrong.

This makes rollback essentially instant. If the new version misbehaves in the first few minutes, flipping the router back is a single change, and the old environment is already running and serving correctly.

The cost is easy to understand. Two production environments cost roughly twice as much as one. For some teams, that double bill is acceptable for the rollback guarantee. For others, it’s the reason blue-green is not adopted. The database is also a problem. Most teams can’t afford two production databases, so the same database serves both environments, which means schema changes have to work for both versions simultaneously. This is part of why expand-contract migrations exist, and we’ll look at it in a later section. User sessions in progress can also break when traffic switches, since the new environment doesn’t know about sessions established in the old one.

Blue-green is often described as a zero-downtime deployment, but rolling deployment also delivers zero downtime. The defining benefit of blue-green is the speed of rollback.

Canary Deployment

Canary deployment removes the all-at-once switch entirely. Instead of moving 100% of traffic to the new version at the moment of release, a canary rollout sends a small fraction (often one percent, sometimes less) to the new version while the rest stays on the old. The team then watches what happens to that fraction, and if metrics look healthy after a baking period, the percentage increases (one percent, then ten, then fifty, then everyone). If the metrics look bad, the small percentage gets rolled back, and the bulk of users never notice anything was wrong.

The reason teams choose canary deployment is the realization that pre-production testing has limits. Some bugs only show up under real production traffic with real production data and real production scale, and the only way to find them is to let real users encounter the new version. Canary makes that encounter manageable by limiting the size of the affected user base.

Canary requires something that the previous strategies don’t, which is observability tagged by version. The whole strategy hinges on being able to compare the new version’s error rate, latency, and behavior to the old version’s, in real time, on a small slice of traffic. A team that tries to canary without per-version metrics is exposing one percent of its users to potentially broken code with no way to detect that it’s broken. In that case, the strategy collapses into a slow leak.

Canary also requires defining success in advance. Before the rollout starts, the team has to agree on what “healthy” looks like and what triggers an automatic or manual rollback. Without those criteria, the rollout becomes a negotiation under pressure, which is exactly when teams make bad decisions.

Feature Flags

The strategies covered so far have all been about managing how new code moves across the fleet.

Feature flags introduce a different idea. Deploying code and releasing it to users can be separate events, and most of the strategies that follow in this article rely on that separation in one way or another.

A feature flag is a conditional check in the code that decides, at runtime, whether a particular block runs. The condition isn’t compiled into the code. It’s read from a configuration source like a database, a config service, or a vendor product that can be changed without redeploying anything. The simplest version is a boolean. If the flag is on, the new code path runs. If it’s off, the old one does.

With feature flags in place, code can be deployed in the dark. The flag stays off in production, no user encounters the new behavior, and the team can verify that the deploy was successful in the narrow sense of “the new code is on the servers and isn’t crashing them.” Later, the flag flips. The flag might be enabled first for internal employees, then for one percent of users, then for everyone in a specific country, then for all users. Each flip is a release. None of them requires a deploy.

Feature flags also enable kill switches, where a problematic feature can be turned off instantly without rolling back the deploy that introduced it. They enable user-segment targeting, where different users see different versions of the product. They are what make A/B testing and canary releases practical in many real systems.

The cost happens at scale. Every flag is a branch in the code. Ten flags create a hundred potential combinations of code paths, most of which were never tested together. A team running a hundred flags in production has a system that is hard to reason about, where a configuration change can produce behavior that no engineer has ever seen. Over time, this accumulates into tech debt, where flags created for a rollout years ago are never cleaned up and remain readable from config. Forgotten flags have caused real incidents.

Discipline matters here. Flags need owners, naming conventions, sunset dates, and observability of their own state.

Dark Launches

A dark launch is what happens when new code runs in production without any user seeing its output. The new code is deployed, the relevant code path executes on real requests, but its results are discarded or only logged. Often, the old code path runs alongside it, so each request is processed twice, once by the existing system and once by the new one, and the two outputs can be compared.

The point of running code this way is to test it against real production conditions without exposing users to its output. A team replacing a search ranking algorithm can run the new ranker against real queries, log what it would have returned, and compare those results to what the existing ranker returned. The user only ever sees the existing ranker’s output, while the new ranker gets weeks or months of real-world validation under real load before any user is exposed to it.

This approach is most useful for backend changes where the system’s behavior, not the user’s experience, is what’s being validated. The kinds of changes that fit are new algorithms, new database queries, new computation paths, or new versions of an internal service. What dark launches can’t validate is anything user-facing, because by design, no user encounters the change.

Shadow Traffic

Shadow traffic is the same idea applied at a different layer. Instead of running two code paths inside one service, the routing layer duplicates each incoming request and sends one copy to the existing service and another copy to a new service that the team is validating. The existing service processes the request normally and returns its response to the user. The new service processes the same request, but its response goes nowhere. The user has no idea that anything was duplicated.

Shadow traffic fits situations where an entire system is being rewritten or replaced rather than just a code path. Common cases include a new pricing service, a new recommendation backend, or a new payments processor. The new system gets real production traffic, with real production patterns, at real production volumes, while the existing system continues to serve users.

The main concern is side effects.

If the new service writes to a database, sends emails, charges credit cards, or otherwise affects the world, those effects don’t get magically discarded along with the response. Shadow traffic works cleanest for operations that only read data, or for ones that can safely run more than once without changing the outcome (called idempotent operations). For anything that writes, the new service needs to either point at a separate database or have its writes carefully gated, which adds operational complexity.

A/B Testing

A/B testing reuses the deployment infrastructure to answer a question that the previous strategies don’t.

Canary asks whether the new version is healthy in system terms, like errors, latency, and crashes. In contrast, A/B testing asks whether users behave differently when they encounter the new version. Do they convert more, click more, retain better, or finish the checkout flow more often?

See the diagram below:

The mechanism looks similar to canary deployment, with traffic split between two versions and metrics collected from each. What differs is the measurement target. Canary measures the system, while A/B testing measures the users. And because user behavior is noisy, A/B tests need statistical significance to produce trustworthy answers, which means they run for days or weeks rather than minutes or hours.

Confusing A/B testing with canary deployment is common and leads to bad decisions. A team that runs an A/B test thinks it has used canary deployment, but its observability is set up to measure conversion rate rather than error rate, so it can miss real production incidents. A team that runs a canary thinks it has measured product impact, but a half-hour bake period can’t possibly produce a statistically meaningful behavioral signal. The mechanism is shared, but the questions being answered are not the same.

Expand-Contract

Every strategy covered so far makes one assumption. Code can be deployed and rolled back freely, with the system returning cleanly to its previous state if needed. This holds for most code changes. It does not hold for database schema changes, and that is where most of the strategies above quietly fail in practice.

Consider a schema change that drops a column. The deploy includes the schema change and the code change that no longer uses the column. The deployment completes. An hour later, something goes wrong with the new version, and the team wants to roll back. The old code expects the column to be there, but the column is gone. The rollback breaks the system worse than the bad deploy did.

The pattern that handles this is called expand-contract, and the name describes the steps. To change a schema safely, the change is spread across multiple deploys:

  • First comes the expand phase, where new schema elements are added alongside the old ones without removing anything.

  • Then the code is deployed that reads from the old structure but writes to both old and new, which is called dual writing.

  • Then the data is backfilled, copying old values into the new structure.

  • Then the code is deployed that reads from the new structure, but still writes to both.

  • Only once all reads and writes are using the new structure is the contract phase safe, where the old structure is finally removed.

This might sound laborious. A single logical change like renaming a column becomes four or five deploys spread over hours, days, or weeks. The cost is definitely there, and the benefit is that every intermediate deploy is independently rollback-safe. There is no point in the sequence where rolling back the most recent deploy leaves the system in a state it can’t run in.

Tradeoffs

Three tradeoffs show up across all these strategies.

  • Speed versus safety: More sophisticated strategies are safer in terms of blast radius and recoverability, but they are slower. A canary rollout that runs for forty-eight hours at each stage adds days to every release cycle. A blue-green deployment requires the new environment to be fully validated before the switch. Feature-flagged rollouts can sit at five percent for weeks while a team gathers confidence.

  • Infrastructure and operational cost: Sophisticated strategies cost real money and real engineering time. Blue-green roughly doubles production infrastructure spend. Canary requires per-version observability that itself has to be built and maintained. Feature flag systems need their own tooling, their own observability, and their own discipline. Shadow traffic infrastructure has to mirror real production traffic patterns without doubling user-visible load.

  • Cognitive load that accumulates: Each strategy a team adds becomes something to maintain, to monitor, and to debug when it misbehaves. Feature flags can accumulate into flag debt. Mixed-version states during rolling deploys produce bugs that are hard to reproduce because they only happen during certain rollout windows. Expand-contract migrations require discipline to complete cleanly, and incomplete contractions where a new schema is added but the old schema is never removed accumulate as another form of debt.

The practical reality is that strategies combine. A real delivery setup might use feature flags inside canary rollouts inside blue-green environments with expand-contract migrations underneath all of it. These strategies are like building blocks.

The key takeaway from all of this is that the right strategy is bounded by what a team can actually run safely, given its observability, its infrastructure, and its discipline.

Summary

In this article, we’ve looked at deployment strategies in detail. Here are the key learning points in brief:

  • Big-bang deployment stops the old version and starts the new one. It is rational for simple systems with maintenance windows, even though a bad deploy affects everyone at once.

  • Rolling deployment replaces instances a few at a time so traffic never stops, but the gradual rollout lets subtle bugs poison the entire fleet before anyone notices.

  • Blue-green keeps two identical production environments and switches traffic between them. Doubled infrastructure cost buys instant rollback, which is the real benefit.

  • Canary deployments send a small fraction of traffic to the new version and increase it as metrics confirm health. Without per-version observability, the strategy is just exposing fewer users to broken code more slowly.

  • Deploying code and releasing it to users are two separate events, and treating them as the same is the most common gap in how deployment gets taught.
    Feature flags are conditional code paths controlled by external configuration, and they are the operational foundation that makes separating deploy from release practical at scale.

  • Dark launches run new code paths invisibly inside an existing service, often alongside the old code, validating system behavior without showing anything to users.

  • Shadow traffic duplicates production requests to a separate new service whose responses are discarded, validating an entire system under real load while risking side effects from writes that leak through.

  • A/B testing reuses the same traffic-splitting machinery as canary but measures user behavior instead of system stability, which means tests run longer because behavioral signals need statistical significance.

  • Schema changes break the assumption that code can be rolled back freely, and the expand-contract pattern handles this by spreading each logical schema change across multiple deploys.
