The Must-Know Fundamentals of Distributed Systems

Every Google search, Netflix stream, and bank transfer relies on distributed systems where multiple computers work together to accomplish tasks impossible for a single machine. Understanding how these systems handle communication, failures, and coordination is becoming essential for modern software developers.

The fundamental challenge that makes distributed systems different is partial failure. In single-computer programs, everything typically crashes together. In distributed systems, some components can fail while others continue operating. For example, a database might crash while web servers keep running, or network connections might fail while both services remain healthy.

This creates ambiguity. When we send a request and receive no response, we cannot determine what happened.

  • Did the request never arrive?

  • Did the server process it, but crash before responding?

  • Did the response get lost?

Every concept in distributed systems addresses some aspect of this challenge.

In this article, we will look at five foundational topics around distributed systems: how computers communicate across networks, the protocols enabling reliable communication, how remote procedure calls abstract complexity, strategies for handling failures, and why time synchronization presents unique challenges.

How Computers Communicate

Networks exist in two configurations.

  • Local Area Networks (LANs) connect computers within data centers, offering speeds from 100 Mbps to 10 Gbps with under 5 milliseconds of latency.

  • Wide Area Networks (WANs) span continents with higher latency (50-300 milliseconds) and more failure points.

The Internet is the largest WAN. We cannot make New York and Sydney communicate faster than the speed of light allows, imposing a minimum 160 millisecond round-trip time.

Most systems use both. Services within data centers use LANs, while multiple data centers connect via WANs. Bandwidth measures how much data a connection can carry (for example, 100 Mbps to terabits per second). Latency measures travel time, including propagation delay (speed of light), transmission delay (pushing bits onto wire), and processing delay (router forwarding time).

The main insight to remember here is that latency is often the bigger bottleneck. A typical image download involves 5 milliseconds of transfer but 100-150 milliseconds waiting for the first byte. We can add bandwidth, but cannot beat physics. This is why systems place servers near users using techniques like CDNs.

The Internet uses packet switching. Data breaks into packets (up to 1,500 bytes) with source and destination addresses. Packets travel independently, taking different routes through the network. Routers examine each packet’s destination and forward it accordingly. A message might hop through 10-20 routers.

The advantage is that if one route fails, packets can take alternate paths. The cost is complexity. Packets can arrive out of order, get duplicated, or disappear. This unreliability requires protocols like TCP for reliability.

The Protocols That Enable Reliability

TCP, or Transmission Control Protocol, is one of the most important protocols for communication in distributed systems. It is connection-oriented and reliable. It also establishes connections through a three-way handshake: client sends SYN, server responds with SYN-ACK, client confirms with ACK. This requires one full round-trip before data transfer.

TCP provides reliability through sequence numbers (allowing reordering and duplicate detection), acknowledgments (confirming receipt), retransmission (resending on timeout), checksums (detecting corruption), and flow control (preventing receiver overwhelm).

However, TCP has a limitation called head-of-line blocking. Since it guarantees order, if Packet 1 is lost but Packet 2 arrives, the application cannot receive Packet 2 until Packet 1 arrives. The entire stream freezes.

In contrast, UDP is connectionless and unreliable. It sends packets without connections, acknowledgments, or retransmission. UDP uses an 8-byte header versus TCP’s 20+ bytes and skips handshakes. Lost packets stay lost.

We should use UDP when speed requirements trump reliability needs, such as during DNS lookups, video calls (dropped frames beat frozen video), gaming (responsiveness over perfection), and streaming. Many UDP applications add custom reliability when needed.

Different applications have different needs. For example, bank transfers need reliability, and extra seconds are acceptable. Video calls need real-time delivery, and skipping frames is acceptable to complete pausing.

QUIC builds TCP-like reliability on UDP. It supports multiple independent streams in one connection. If Stream A loses a packet, Stream B continues unaffected, preventing single packet losses from stalling entire page loads. QUIC combines transport and cryptographic handshakes, allowing 0-RTT data transmission for repeat connections. QUIC uses connection IDs rather than IP addresses, so mobile users switching networks maintain their sessions.

For reference, HTTP3 uses QUIC under the hood.

Remote Procedure Calls

Remote Procedure Calls (RPC) let developers call remote functions as if they were local. Arguments are packaged, shipped to a server, executed, and the results are returned as if the function ran locally.

Without RPC, we manually design message formats, handle serialization across platforms, manage connections, implement retry logic, and handle failures. RPC frameworks handle this through marshalling, which converts data from internal format to network format.

A client-side stub acts as the local representative. It marshals parameters, sends requests, waits for responses, unmarshalls results, and returns them. The server-side skeleton receives requests, unmarshalls them, invokes the actual function, marshalls results, and sends responses.

SOAP emerged in the late 1990s using XML for interoperability between languages. It provided robust security and transactions, becoming standard in banking and telecommunications. However, XML verbosity and specification complexity made it heavy.

On the other hand, REST became popular in the 2000s. It used standard HTTP methods (GET, POST, PUT, DELETE) on resources instead of remote procedures. Each request contains all the needed information. Servers hold no session state. REST uses lightweight JSON and became the standard for public APIs due to simplicity and loose coupling.

gRPC uses Protocol Buffers (binary format) for efficiency. Messages are many times smaller than JSON, where serialization is 10 times faster with no text parsing. Built on HTTP/2, gRPC enables bi-directional streaming and multiplexing. It is now standard for internal microservice communication, while REST remains standard for external APIs.

However, location transparency can be misleading. Network calls can take milliseconds versus nanoseconds, fail in unique ways (timeouts, partial failures), and never guarantee delivery. Modern practices recommend making developers aware of remote calls rather than hiding them.

Handling Partial Failures

In single computers, failures are all-or-nothing. However, in distributed systems, some parts fail while others continue. The most critical aspect is ambiguity. When we send a request without getting a response, we cannot determine what exactly happened: the request never arrived, or the server crashed mid-processing, or the response got lost.

Therefore, every network call needs a timeout to prevent threads from waiting forever and resources from being exhausted. However, timeouts can also create ambiguity. They tell us something might be wrong, but not what. The operation may not have been received, may have completed without acknowledgment, or may still be executing. This is why retrying after a timeout is dangerous without proper safeguards.

When transient errors occur (network blips), retrying is correct. However, naive retries can bring down systems. If a database slows and 10,000 clients retry immediately, the load increases instantly, preventing recovery. Exponential backoff waits increasingly longer between retries (1s, 2s, 4s, 8s), reducing call rate during outages. Also, Jitter adds randomness to wait times, preventing synchronized retry waves that create traffic spikes.

Retries also introduce a new risk of executing operations twice. If we retry a payment charge after a timeout, the first call might have succeeded, causing a double charge. To handle this, we need to build idempotent operations.

An operation is idempotent if applying it multiple times equals applying it once. For reference, HTTP GET, PUT, and DELETE are inherently idempotent. Deleting a user ten times produces the same state. However, the POST method is typically not idempotent. This means creating payments multiple times creates multiple records.

For example, consider withdrawing $100 without idempotency: the request is sent, the server processes ($1000 to $900), and the response is lost. We retry and server processes again ($900 to $800). The customer ends up losing $100.

However, with idempotency, we send “Withdraw $100 with request-id ABC123.” Server processes and stores ABC123 successfully. However, the response is lost, and we retry with ABC123. The server sees that ABC123 has already been processed and returns the cached result.

Idempotent operations can be implemented using idempotency keys:

  • Client generates a unique ID (UUID) before the request.

  • Client includes it in the header (Idempotency-Key: abc-123-xyz).

  • Client retries with the same key if the request fails.

  • Server checks if the key exists in storage.

  • If not found, the server processes the request and stores the key.

  • If found, the server returns the stored response.

In message systems, these translate to delivery guarantees.

  • At-most-once means messages delivered zero or one times (may be lost but never duplicated). Simple, low-latency, suitable for non-critical data like metrics.

  • At-least-once guarantees arrival but allows duplicates (we retry until acknowledgment). This requires receivers handle duplicates.

  • Exactly-once is theoretically impossible in distributed systems. We can never be 100% certain a message was received because acknowledgments could be lost.

The practical solution is to use at-least-once delivery with idempotent operations, which achieves exactly-once behavior.

The Problem of Time in Distributed Systems

In centralized systems, the operating system’s clock is the truth. However, in distributed systems, every server has its own clock, and these clocks drift apart due to hardware imperfections.

Computer clocks use quartz crystal oscillators. Temperature fluctuations, manufacturing variations, and aging cause crystals to run fast or slow. Typical drift is 10-50 parts per million, corresponding to 1-4 seconds per day. After one day, two computers might differ by hundreds of milliseconds. After a month, they could be seconds apart.

Operating systems maintain two clock types.

  • Time-of-day clocks (wall clocks) show the current time in hours, minutes, and seconds. They can jump forward or backward due to NTP adjustments, daylight saving time, manual corrections, and leap seconds.

  • Monotonic clocks count time forward from an arbitrary point (like system boot). The absolute value is meaningless; they measure intervals. They never jump backward and are unaffected by NTP adjustments.

Network Time Protocol (NTP) synchronizes clocks by querying time servers. It uses a hierarchical architecture:

  • Stratum 0 devices are high-precision references (atomic clocks, GPS).

  • Stratum 1 servers connect directly to them.

  • Lower strata synchronize with higher ones.

NTP calculates the offset by exchanging timestamps and measuring round-trip delay, then gradually adjusts the clock.

Typical accuracy is 10-100 milliseconds over the internet and under 1 millisecond on well-configured LANs. But asymmetric paths, system delays, and congestion can cause 100+ millisecond errors. Perfect synchronization is physically impossible, and network delays are unpredictable. We cannot read two clocks “at the same instant” across a network. Even after synchronization, clocks immediately drift apart.

Given two events on different machines, we often cannot definitively determine which happened first. If clocks could differ by 50 milliseconds and events are timestamped 30 milliseconds apart, we cannot be certain of ordering.

However, we often do not need actual time, but just the order of causally related events. The happened-before relation captures this. If events A and B occur in the same process with A first, A happened-before B. If A sends a message and B receives it, A happened-before B. If neither a happened-before relationship exists, events are concurrent with no meaningful ordering.

Lamport clocks implement this with logical counters.

Each process maintains a counter, incrementing before events and updating based on received messages. This guarantees that if A happened before B, then the timestamp of A is less than the timestamp of B, all without synchronized physical clocks.

Summary

In this article, we have looked at some fundamental aspects of Distributed Systems in detail. Here are the key learning points in brief:

  • Partial failure distinguishes distributed systems from single-machine programs: some components can fail while others continue operating, creating ambiguity that traditional software never encounters.

  • Latency, not bandwidth, typically bottlenecks distributed systems: we can add more bandwidth by upgrading hardware, but physics limits how fast data travels across distances, making geographic placement crucial.

  • TCP provides reliability through sequence numbers, acknowledgments, and retransmission, but suffers from head-of-line blocking where a single lost packet freezes the entire stream until retransmitted.

  • UDP trades all reliability guarantees for speed by eliminating handshakes, acknowledgments, and retransmission, making it ideal for real-time applications where old data becomes useless.

  • QUIC solves TCP’s head-of-line blocking by supporting multiple independent streams over UDP, allowing one stream to continue processing while another waits for lost packets.

  • Remote procedure calls hide network complexity but can mislead developers: network calls fundamentally differ from local calls in latency (milliseconds versus nanoseconds) and failure modes (timeouts, partial failures).

  • RPC technologies evolved from verbose SOAP (XML-based enterprise standard) through simple REST (HTTP methods on resources with JSON) to efficient gRPC (binary Protocol Buffers on HTTP/2 for internal microservices).

  • Timeouts are essential for preventing resource exhaustion from hung connections, but they create inherent ambiguity: we cannot determine whether an operation failed, succeeded, or is still processing.

  • Exponential backoff prevents retry storms by waiting increasingly longer between retries (1s, 2s, 4s, 8s), while jitter adds randomness to prevent synchronized waves of retries that overwhelm recovering services.

  • Circuit breakers protect distributed systems from cascading failures by failing fast when downstream services are unavailable, entering an open state after error thresholds, and testing recovery in a half-open state.

  • Idempotent operations produce the same result when applied multiple times: GET, PUT, and DELETE are naturally idempotent, while POST typically requires explicit idempotency handling.

  • Idempotency keys make retries safe by allowing servers to recognize duplicate requests: clients generate unique IDs before requests, servers check for previously processed keys, and return cached results for duplicates.

  • At-least-once delivery guarantees message arrival but allows duplicates, while exactly-once delivery is theoretically impossible because acknowledgments could be lost, making certainty unattainable.

  • The practical approach combines at-least-once delivery with idempotent operations to achieve effective exactly-once semantics: we do not need impossible guarantees if operations handle duplicates gracefully.

  • Computer clocks drift 1-4 seconds per day due to quartz crystal imperfections affected by temperature, making perfect time synchronization across distributed systems physically impossible.

  • Network Time Protocol synchronizes clocks with 10-100 millisecond accuracy over the internet, but network delays are unpredictable: we cannot read two clocks “at the same instant” across a network.

  • Lamport’s logical clocks capture event ordering without requiring synchronized physical time: if event A happened-before event B, then A’s timestamp is less than B’s timestamp, using counters instead of wall clocks.
