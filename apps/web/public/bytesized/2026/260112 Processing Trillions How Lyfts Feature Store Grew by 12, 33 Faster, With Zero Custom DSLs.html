TL;DR

Lyft’s Feature Store serves 60+ production use cases and grew caller count by 25% last year. They cut P95 latency by a third while handling over a trillion additional R/W operations. The secret wasn’t fancy tech—it was treating ML infrastructure like a product with actual users who have better things to do than learn your system.

🎯 The Problem Nobody Talks About

Here’s what kills most ML platforms: the feature engineering tax.

Data scientists write a killer model. It works great in notebooks. Then they need to:

  • Rewrite feature logic for production (different language, different compute)

  • Debug why training features don’t match serving features

  • Wait 3 sprints for platform team to provision infrastructure

  • Maintain two separate codebases that drift apart

Six months later, the model’s still not deployed and everyone’s moved on to the next fire.

Lyft decided this was unacceptable. When you’re running a marketplace where every ML improvement directly impacts revenue, you can’t have your ML engineers stuck in infrastructure hell.

🏗️ Architecture That Doesn’t Get In The Way

The Three Feature Families

Lyft split their world into batch, streaming, and online. Not revolutionary, but the execution matters.

Batch features (the workhorse):

  • Customer writes SparkSQL query + simple JSON config

  • Python cron generates production Airflow DAG automatically

  • DAG handles compute, storage, quality checks, discovery—everything

  • Data lands in both Hive (offline training) and DynamoDB (online serving)

Streaming features (the real-time stuff):

  • Flink apps read from Kafka/Kinesis

  • Transform data, add metadata

  • Sink to spfeaturesingest service

  • Service handles serialization and writes to online store

Online serving (dsfeatures):

  • DynamoDB as source of truth

  • ValKey (Redis fork) write-through cache on top

  • OpenSearch for embeddings

  • Go and Python SDKs expose full CRUD

The smart part? Whether you write features via batch DAG, streaming app, or direct API call, they all land in the same online store with identical metadata. No “training/serving skew” headaches.

The Part That Actually Matters

Most feature stores fail because they’re too clever. Lyft succeeded because they made everything stupidly simple:

For feature creation:
SparkSQL query + JSON config. That’s it.

json

{
  "owner": "pricing-team",
  "urgency": "high",
  "refresh_cadence": "daily",
  "features": {...}
}

sql

SELECT 
  user_id,
  avg(ride_cost) as avg_ride_cost_30d
FROM rides
WHERE dt >= date_sub(current_date, 30)
GROUP BY user_id

No YAML hell. No custom DSLs. Just SQL and basic metadata.

For feature retrieval:
SDK method calls. Get() or BatchGet(). Returns data in whatever format your service speaks.

They optimized for the 90% use case: SQL-proficient engineers who want to ship fast and move on.

💡 What They Got Right

1. Kyte: Local Development That Works

Lyft built Kyte, their homegrown Airflow local dev environment. The Feature Store team added a CLI on top:

  • Validate configs before pushing

  • Test SQL queries immediately

  • Run full DAG locally

  • Backfill with confidence

This is huge. Most platform teams ship tooling that only works in prod, then wonder why adoption sucks. If developers can’t iterate locally, they won’t use your platform.

2. Amundsen Integration: Discovery or Death

Generated DAGs auto-tag feature metadata in Amundsen (Lyft’s data discovery tool). Engineers can search for features before building new ones.

Why this matters: Feature duplication is silent technical debt. Someone builds avg_ride_cost_30d. Three months later, another team builds average_cost_last_month. Same data, different names, double the compute costs.

Discovery prevents this. It also prevents the “I didn’t know that existed” problem that makes new engineers feel stupid.

3. Treating Platform Like Product

The team made hard choices:

What they cut:

  • HiveQL support (Pareto principle—focus on core use cases)

  • Redshift compatibility (niche, low value)

  • Custom orchestration (migrated from Flyte to managed Astronomer)

What they added:

  • Golang SDK (top customer request)

  • Offline Python SDK (normalize access patterns)

  • Staging environment (test before prod)

  • Data contracts (enforce SLAs)

They didn’t try to be everything to everyone. They identified their actual users (software engineers + ML modelers), learned what those users needed, and ruthlessly prioritized.

4. Operational Excellence Isn’t Optional

Generation transparency:

  • Better monitoring on failed DAGs

  • Ownership tracking

  • Confidence to deprecate unused features

Retrieval reliability:

  • Upgraded cache to ValKey

  • Stripped unnecessary payload fields

  • Right-sized EKS pods to reduce Redis connection issues

  • Hardened retry/timeout policies

  • Cranked up cache TTL

Result: P95 latency down 33%. Customer support threads way down. On-call engineers actually sleeping.

⚠️ The Stuff That’s Hard

AWS Is Your Frenemy

DynamoDB doesn’t SLA latency. Neither does most AWS infrastructure. You can have perfect code and still get hammered by tail latencies from your backing store.

Lyft’s solution: be as lean as possible. Smaller payloads, smarter caching, aggressive TTLs. You can’t control AWS, but you can minimize exposure.

Streaming Is Non-Trivial

Batch features are easy. Daily refreshes, predictable compute, well-understood failure modes.

Streaming features are chaos. Out-of-order events, state management, exactly-once semantics, backpressure handling. Lyft is investing heavily here (new headcount, new abstractions) because demand is growing faster than their ability to make it easy.

The RealtimeMLPipeline interface is their attempt to abstract the pain. Too early to tell if it works, but at least they’re trying.

Embeddings Are Different

You can’t just shove embeddings into DynamoDB and call it done. They need specialized indexing (OpenSearch), different retrieval patterns, different performance characteristics.

Lyft added this support recently. It’s generating “significant interest” which probably means everyone wants to build RAG apps now.

🎬 What You Can Steal

If you’re building ML infrastructure:

  1. SQL + JSON > custom DSLs. Your users already know SQL. They don’t want to learn your snowflake syntax.

  2. Auto-generate production workflows. Don’t make people write Airflow DAGs by hand. Generate them from configs. Include monitoring, quality checks, metadata—everything.

  3. Local dev is non-negotiable. If engineers can’t test locally, they’ll cowboy commits to prod and blame your platform when things break.

  4. Unified metadata across batch/streaming/online. Training/serving skew is a deployment killer. Same features, same metadata, same retrieval interface.

  5. Discovery > documentation. Nobody reads docs. People search for existing solutions. Make search work.

  6. Cut aggressively. Supporting niche use cases kills your ability to execute on core ones. The Pareto principle is real.

  7. Monitor what matters. Failed DAGs, data lateness, retrieval latency, cost trends. If you’re not measuring it, you can’t improve it.

If you’re not building ML infrastructure:

The pattern still applies. Platforms succeed when they:

  • Solve real user pain (not imaginary future needs)

  • Make the simple case trivial (not the complex case possible)

  • Invest in DX as much as features

  • Treat internal tools like external products

The Real Story

Lyft’s Feature Store grew 12% in feature count and 25% in distinct callers, year-over-year, during a period where they actively deprecated features.

That’s not normal. Most platforms fight for adoption. Lyft’s problem is managing demand.

The trillion additional read/write operations aren’t because they got lucky. They’re because the platform team understood that infrastructure adoption is a people problem, not a technology problem.

Nobody cares how elegant your architecture is if it takes three weeks to ship a feature. Nobody cares about your cutting-edge tech if the getting-started guide is incomprehensible.

Lyft built something that got out of the way and let people ship. That’s why it worked.
