How Etsy Reduced Page Load Time to 0ms

Byte-Sized Design

Paid

Etsy shipped a performance improvement so dramatic that 40% of their users now see product pages load in essentially zero milliseconds. No infrastructure overhaul. No rewrite. Just a clever use of browser prediction and a 15-line JSON config.

If you’re thinking “prefetching is old news,” I’ve got news: you haven’t seen the Speculation Rules API yet.

The 200ms Window

The traditional web flow is brutally wasteful. User hovers over a product link. User’s brain decides this looks interesting. User moves cursor to click. Click event fires. Browser initiates request. DNS lookup. TCP handshake. TLS negotiation. HTTP request. Server processing. Response headers. HTML starts streaming. Parser kicks in. More requests for CSS, JavaScript, images.

The entire time between hover and click—typically 200-500ms—the browser just sits there. Your user has already made their decision. The machine is waiting for permission.

This is the opportunity Etsy exploited.

What Makes Speculation Rules Different

The old <link rel=”prefetch”> has been around forever. It downloads resources into HTTP cache. Nothing revolutionary. The Speculation Rules API does something more interesting: it caches prefetched pages in memory, not just HTTP cache. This matters because memory access is orders of magnitude faster than disk.

More importantly, SRA gives you a declarative way to specify when prefetching happens. No bespoke JavaScript watching hover events and managing state. Just JSON:

json

{
  “prefetch”: [{
    “where”: {
      “href_matches”: “/listing/*”,
      “selector_matches”: “[data-sr-prefetch=’1’]”
    },
    “eagerness”: “moderate”
  }]
}

The data-sr-prefetch attribute lets Etsy opt in specific links. The href_matches pattern ensures only listing pages get prefetched. And “moderate” eagerness means the browser waits 200ms after hover before initiating the prefetch—filtering out accidental hovers while still giving a meaningful head start.

The Implementation Reality

In terms of code changes, this was trivial. The search team added one script tag to the search results page. Done.

But that’s where the easy part ended.

They spent most of their time dealing with edge cases that modern web applications accumulate like barnacles. Product videos that swallow hover events. Analytics systems that conflate “page requested” with “page viewed.” Redirects. Cookie-setting behavior. Link hrefs that get mutated by JavaScript.

The video problem turned out to be particularly gnarly. Listings on Etsy often include product videos that autoplay on hover. These use browser-native video controls, which live in shadow DOM. Shadow DOM is isolated from the regular DOM, which means hover events on the video element never bubble up to the link element wrapping it. The prefetch never triggers.

Their solution was creative: overlay a transparent div on top of the video for exactly 200ms. Long enough to let the hover event register and trigger prefetching. Then remove the div so the browser’s video controls (play, pause, right-click menu) work normally. Hacky? Absolutely. But it works.

The redirect situation was more forgiving than expected. If your link goes through a redirect (/listing/redirect → /listing/123), the browser follows the redirect during prefetch and caches the final page. When the user clicks, the browser still goes through the redirect flow, but if you’ve set appropriate cache headers, the redirect response itself is cached. So even though redirects are still a performance anti-pattern, they don’t break prefetching.

Mutating hrefs turned out to be a silent killer. Some sites append tracking parameters on hover for analytics: link.html becomes link.html?source=search. This completely breaks prefetching. The browser starts downloading link.html, JavaScript changes the href, browser sees a different URL and abandons the prefetch. Worse, the failed prefetch attempt still counts toward the browser’s limit of two concurrent speculative loads, potentially evicting a successful prefetch from memory. Don’t do this.

The Analytics Nightmare

This was the hardest part of the entire project. Not the prefetching mechanism—that was 15 lines of JSON. The hard part was ensuring their analytics remained accurate.

Every analytics system on the planet was built with one fundamental assumption: if the server constructed and sent an HTML page, a human saw it. Speculation Rules shatters this assumption. A prefetch request looks identical to a real navigation from the server’s perspective. Same HTTP request. Same response. But the user might never see the page.

For Etsy, this wasn’t just about pageview counts. Their entire search ranking pipeline depends on accurate view_listing events. These events contain listing ID, user ID, search context, and more. They feed into search training models, recently viewed lists, recommendation systems. If prefetch requests started firing these events, they’d poison their entire data pipeline with phantom views.

Their solution was architectural. They built a system to cache the payload of all events within a request. During prefetch (detected via the Sec-Purpose: prefetch header), events get cached but not fired. They moved event logging to the destination page’s JavaScript bundle, deferring execution until after page “activation”—the moment when the user actually navigates and the prefetched page becomes visible. The browser provides APIs for this: document.prerendering tells you if the page is currently being prerendered/prefetched, and the prerenderingchange event fires when activation happens.

This required coordination with analytics partners. Third-party scripts had to be updated to respect the prerendering state. Internal event logging had to be refactored. Analytics dashboards had to be verified for accuracy. This took more engineering time than anything else in the project.

The Memory Cache Game

Here’s something that surprised the team: the browser only keeps two prefetched pages in memory at once. Prefetch a third and the first gets evicted to make room.

This creates a race condition. User hovers over listing A (prefetch starts). User hovers over listing B (prefetch starts). User hovers over listing C (prefetch starts, A evicted from memory). User clicks listing A—cache miss.

The HTTP cache might still have listing A if it’s cacheable, but memory cache is gone. And here’s the kicker: Etsy’s listing pages are non-cacheable due to their complexity. Personalization, inventory status, dynamic pricing—these pages can’t be cached in the traditional sense.

So they built a workaround. When the server detects a prefetch request (via Sec-Purpose: prefetch header), it makes the page cacheable for five minutes. Even if the page gets evicted from memory cache due to the two-page limit, the HTTP cache preserves it. This five-minute window is actually a browser guarantee prefetched pages stay in memory cache for up to five minutes before normal caching rules apply.

This detail matters because it changes the cost-benefit calculation. Without HTTP cache fallback, the benefits of prefetching would collapse for any user who hovers over more than two links before clicking. With it, you get memory cache speed for recent hovers and HTTP cache speed for older ones.

The Eagerness Decision

Speculation Rules offers four eagerness levels, and choosing the right one is a product decision disguised as a technical one.

“Immediate” eagerness prefetches every eligible link as soon as the page loads. This would hammer Etsy’s servers with prefetch requests for dozens of listings the user will never click. Wasteful and expensive.

“Conservative” eagerness waits until mousedown—the moment between pressing the mouse button and releasing it to complete the click. This gives you maybe 50-100ms of head start. Barely worth the complexity.

“Eager” triggers on any hover, no delay. Better than immediate, but still generates a lot of wasted prefetches for accidental hovers or users scanning the page.

“Moderate” waits 200ms after hover. This is the sweet spot. It filters out accidental hovers and quick scans while still giving meaningful time for the browser to download the page before the click happens. Most intentional hovers last well over 200ms—users look at the product image, read the title, check the price. That’s your window.

Etsy chose moderate and accepted the cost: one in 14 prefetches was actually used. A 7% hit rate. Thirteen wasted requests for every successful one. That sounds terrible until you remember that the 14th request—the one the user actually navigates to—loads 24% faster.

The Results (And They’re Absurd)

The performance improvements were dramatic across every metric Etsy cares about.

Time to First Byte at the 75th percentile dropped 23.6%. This is the moment the browser receives the first byte of HTML from the server. For prefetched pages, this happens during the hover—before the user even clicks. When they do click, TTFB is effectively zero because the download already completed.

First Contentful Paint improved 20.7%. This is when the first piece of content appears on screen—text, image, anything. FCP depends on having the HTML, which prefetching provides instantly.

Largest Contentful Paint dropped 21.1%. LCP measures when the largest piece of content finishes rendering—usually the hero image or main product photo. Even though prefetching only downloads HTML (not images), having the HTML ready immediately means the browser can start requesting images sooner.

DOMContentLoaded improved 20.4%. This is when the HTML is fully parsed and the DOM is ready. Again, having the HTML already in cache makes this near-instantaneous.

Page Load (the complete load event) improved 10.6%. This is the smallest improvement because it includes all subresources—images, fonts, CSS, JavaScript which aren’t prefetched. But even here, the head start matters.

The most striking result: roughly 40% of eligible browsers saw their TTFB drop to nearly zero. Not “very fast.” Not “under 50ms.” Essentially zero. The browser already had the HTML before the user clicked.

Look at the cumulative distribution function. Control group (no prefetching) in blue, treatment group (prefetching enabled) in orange. At every percentile, the treatment group is faster. At the median, the improvement is dramatic. At the tail (p95, p99), the improvement is still substantial. The entire distribution shifts left.

The business metrics showed small but measurable improvements. This makes sense listing page views come from many sources (search, shop pages, category pages, direct links), and only search results had prefetching enabled in this experiment. As they roll out prefetching to more surfaces, the business impact should compound.

The Cost Nobody Talks About

Fourteen prefetch requests for every one actual page view. Let’s be honest about what this means.

Every prefetch hits their servers. It runs application code, queries databases, generates HTML, sends bytes over the wire. Thirteen of those requests are wasted. The user never sees the page. Pure overhead.

For a site with Etsy’s traffic, this adds up. More server load. More bandwidth. More database queries. More cache churn. The infrastructure cost is real and linear.

But here’s the thing: the user experience improved by 20%+, and for e-commerce, performance directly drives revenue. Amazon famously found that every 100ms of latency costs them 1% in sales. For Etsy, making product pages load 24% faster isn’t just nice, it’s worth real money.

The calculation is straightforward. What’s the cost of serving 13 extra pages per click? What’s the value of a 20% performance improvement on conversion rate? The second number is bigger. That’s why they shipped it.

What’s Next: The Prerender Upgrade

Prefetching only downloads HTML. The browser still needs to parse it, request subresources (CSS, JavaScript, images, fonts), execute JavaScript, and render the page. All of that happens after the user clicks.

Prerendering goes further. It downloads the HTML, downloads all subresources, executes JavaScript, and fully renders the page in a hidden browser process. When the user clicks, the browser just swaps in the already-rendered page. No parsing, no resource loading, no rendering. True instant navigation.

The performance wins would be massive. First Contentful Paint and Largest Contentful Paint could drop to zero. Time to Interactive could be instantaneous.

But there’s a problem: JavaScript side effects. If your page fires analytics events during load, those would fire during prerender—before the user has seen the page. If your page makes API calls, those happen during prerender. If you set cookies, start video playback, open WebSocket connections, or do anything stateful, it all happens before the user navigates.

This is solvable—the same browser APIs that detect prefetching (document.prerendering, prerenderingchange event) work for prerendering. But it requires careful auditing of all JavaScript on the page to ensure nothing happens until activation.

Chrome is working on something interesting: “prerender-until-script.” Prerendering would stop at the first <script> tag in the HTML but still download all page resources (scripts, styles, images, fonts). You get most of the benefit—resources ready to go—without the risk of JavaScript side effects. For Etsy, this could mean even larger improvements to FCP and LCP without the complexity of managing prerender state.

They’re watching this development closely.
