🚀 Jetflow: Supercharging Cloudflare's Data Ingestion at Petabyte Scale

How Cloudflare's Business Intelligence team built a custom data ingestion framework to handle their growing data needs

As Cloudflare's business has grown, so has the volume and complexity of the data they need to ingest and process. Their existing Extract Load Transform (ELT) solution could no longer keep up, leading the Business Intelligence team to build their own custom data ingestion framework, Jetflow. In total, about 141 billion rows are ingested every day.

TLDR

Jetflow has delivered:

  1. Over 100x efficiency improvement in GB-s: A 19 billion row job that took 48 hours and 300 GB of memory now completes in 5.5 hours using just 4 GB.

  2. >10x performance improvement: Ingestion rates have increased from 60-80,000 rows/sec to 2-5 million rows/sec per database connection.

  3. Extensibility: Jetflow's modular design makes it easy to add support for new data sources like ClickHouse, Postgres, Kafka, and more.

Why this matters

Cloudflare's data is business-critical, powering product decisions, growth planning, and internal monitoring. As their data has grown to petabyte scale, with thousands of tables ingested daily, their previous ELT solution could no longer keep up. Jetflow was born out of the need for a more performant, flexible, and extensible data ingestion framework.

Designing a flexible framework

Jetflow's key design principles were:

  1. Performant & efficient: Able to ingest more data in less time, using less memory and compute.

  2. Backwards compatible: Support migrating individual tables while maintaining Spark compatibility.

  3. Ease of use: Minimal configuration, with no-code options for common use cases.

  4. Customizable: Flexible configuration layer to tune performance and execution.

  5. Testable: Ability to run locally in a containerized environment for thorough testing.

The framework was built around a modular pipeline design, with distinct stages for Consumers, Transformers, and Loaders. This allows each stage to be independently tested and extended.

Optimizing for performance

Two key optimizations in Jetflow's implementation:

  1. Using Arrow for internal data format: Arrow's columnar structure and efficient memory usage reduce serialization overhead and garbage collection costs.

  2. Avoiding row-to-column conversions: Jetflow reads data directly in columnar formats (e.g. ClickHouse's native Block format) and writes Parquet files directly from Arrow columns, avoiding unnecessary conversions.

Case study: ClickHouse ingestion

When testing Jetflow with ClickHouse, the team found that using additional connections didn't improve performance, as ClickHouse was reading faster than the data could be received. By switching to a more optimized database driver (ch-go), they were able to achieve millions of rows per second with a single ClickHouse connection.

Case study: Postgres ingestion

For Postgres, Jetflow bypasses the standard database/sql interface and uses the jackc/pgx driver's internal scan functions to read raw bytes directly. This avoids the performance overhead of reflection-based type conversions, allowing them to achieve nearly 600,000 rows per second per Postgres connection.

What you can learn from this

  1. Understand your unique requirements: Cloudflare's data needs were outgrowing off-the-shelf solutions, requiring a custom framework built around their specific requirements.

  2. Prioritize performance and efficiency: Jetflow's optimizations around memory usage, garbage collection, and data format conversions delivered massive performance gains.

  3. Embrace modularity and testability: Jetflow's pipeline design with independent, testable stages made it easier to extend and maintain.

  4. Don't assume database drivers are optimized for your use case: Profiling and switching to more specialized drivers can unlock significant performance improvements.

  5. Measure and validate your improvements: Cloudflare was able to quantify Jetflow's impact in terms of efficiency (GB-s), throughput (rows/sec), and cost savings.
