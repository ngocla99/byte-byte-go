🌀 From Connector Chaos to Clean Streams: PLAID’s Real-Time Architecture Overhaul

How a team cut Kafka connector costs and scaled cleanly with MongoDB Atlas Stream Processing

The setup

PLAID, a Tokyo-based company behind the real-time analytics platform KARTE, had used MongoDB since 2015. Their product ingests and processes millions of user interaction events across web and mobile in near real-time. The goal was to let customers personalize user experiences with up-to-the-second context.

Originally, they ran MongoDB self-hosted. Then in 2021, they moved to MongoDB Atlas. With Atlas came the ambition to tighten the feedback loop: real-time event ingestion, transformation, and analysis, without the complexity of batch jobs. They wanted everything piped into BigQuery to drive their downstream ML and reporting.

That’s where things started to get messy.

The problem: pipeline sprawl

To stream data from Atlas to BigQuery, they relied on Kafka Connect with custom connectors. The shape of the pipeline looked like this:

MongoDB Atlas → Kafka Source Connector → Kafka → BigQuery Sink Connector

In theory, this worked.

But as the product scaled, reality set in:

  • Connector count exploded. One per collection, per environment (prod, staging, etc.). Multiply that by services and teams.

  • Pricing was linear per connector. Each new pipeline meant higher Confluent Cloud costs.

  • Staging overhead doubled everything. Even feature testing meant provisioning duplicate connectors.

  • Monitoring was brittle. Kafka Connect didn’t integrate cleanly with Datadog. They had limited visibility into lags and throughput.

  • Backfills were clunky. There was no clean way to replay historical data into Kafka or BigQuery without hacks.

PLAID hit a tipping point: they were solving data engineering problems that had nothing to do with their core product.

The solution: stream closer to the source

Instead of solving these problems at the Kafka layer, PLAID looked upstream. In particular, they replaced the Kafka Source Connector with MongoDB Atlas Stream Processing (ASP).

The new architecture:

MongoDB Atlas → Atlas Stream Processing → Kafka → BigQuery

ASP is a native MongoDB feature that watches your Atlas collections and emits change streams through custom stream processors. Think of it like a built-in Kafka producer that speaks Mongo’s aggregation language.

What changed specifically?

  1. Connectors removed

    • ASP replaces Kafka Source Connectors entirely.

    • The sink side (Kafka → BigQuery) stayed the same, so downstream consumers were unaffected.

  2. Cost model flipped

    • Old: Confluent charged per connector task.

    • New: Mongo charges based on Stream Processing Instance (SPI) workers, which scale based on actual processing needs.

    • Result: predictable, lower costs. PLAID didn’t have to think twice about adding new pipelines.

  3. No extra skills needed

    • Stream processors use MongoDB aggregation pipelines—same syntax, same semantics.

    • Engineers didn’t have to learn Kafka Streams, Flink, or some bespoke DSL to write transformations.

  4. One connection per SPI

    • ASP manages connection pooling. Instead of multiple connectors each holding a DB connection, the SPI multiplexes over one.

    • This was a quiet but significant ops win.

  5. Custom backfill tool

    • ASP didn’t support historical replays. So PLAID wrote a custom job to backfill data directly from MongoDB into Kafka when needed.

    • They controlled snapshot sizes and parallelism to avoid reintroducing lag into live pipelines.

  6. Metrics with a sidecar bot

    • ASP doesn’t have native support for Datadog.

    • PLAID built a bot to poll metrics from ASP’s control plane and ship them to Datadog. Alerting, throughput, lag—all restored.

The outcome

  • 3M+ events/day piped into BigQuery

  • 20+ real-time pipelines live in production

  • Zero Kafka source connectors left

  • Costs slashed, ops effort reduced

  • Staging environments are cheap and easy to spin up—no more double-provisioning connectors

And the architecture didn’t drift. They kept using Kafka, BigQuery, and Confluent Cloud. ASP just took over the role Kafka Connect was never good at: pulling change streams from MongoDB at scale.

Why this matters

Most teams building real-time pipelines default to Kafka Connect or DIY producers. It works until it doesn’t—until staging complexity, cost, or monitoring fragility outweigh the benefits.

PLAID didn’t rip and replace their whole stack. They made a targeted substitution: replace connectors with smarter infrastructure closer to the source.

This move follows a clear principle: don’t treat your DB like a dumb data store. If it can stream, let it.

MongoDB Atlas Stream Processing is still young—no native backfill, rough edges in observability—but for a team like PLAID, it removed the biggest pain without introducing new ones.

This kind of swap doesn’t make headlines, but it’s the kind of sharp engineering decision that scales quietly, saves real money, and lets teams stay focused on the product.
