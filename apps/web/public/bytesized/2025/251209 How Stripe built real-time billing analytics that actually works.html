How Stripe built real-time billing analytics that actually works

TL;DR

Stripe’s batch-based billing analytics worked fine when updates could wait 24 hours. By 2024, customers demanded real-time visibility into MRR, churn, and conversions because in fast-moving markets, yesterday’s data loses deals today.

The problem? Subscriptions are stateful nightmares. Every $20 payment needs context from months of history. Batch processing couldn’t scale to sub-hour latency. Preaggregated queries were fast but couldn’t incorporate live data. And letting customers change metric definitions meant reprocessing years of history without breaking real-time ingestion.

The fix? Event-driven streaming with Apache Flink. A brand-new Apache Pinot query engine that aggregates on-the-fly. And a dual-mode system that recalculates history while streaming live updates without the dashboard ever going dark.

🚨 The Breaking Points

Batch Processing Hit a Wall

The old system recalculated subscription state by replaying every event from the beginning of time. Want to know if that June payment was on-time? Re-analyze January through June. For every subscription. Every 24 hours.

This worked until customers started asking: “Why can’t I see this trial conversion that just happened?” Because the batch job won’t run for another 18 hours, that’s why.

Preaggregation Made Queries Fast But Data Stale

Apache Pinot delivered sub-second dashboard queries by precomputing MRR over time in offline batch jobs. Fast responses, but baked-in staleness. Real-time streaming meant throwing out preaggregation—which meant risk of slow, unresponsive queries that would make the dashboard unusable.

Custom Metric Definitions Created a Consistency Nightmare

Customers could tweak MRR formulas (exclude coupons, adjust trial periods, etc.). Great for flexibility. Terrible for streaming systems. Change a definition? Now you need to:

  1. Reprocess 8 years of historical data (hours of computation)

  2. Keep streaming new events using the old definition (can’t stop the world)

  3. Somehow merge them without showing Frankenstein data in the dashboard

There was no playbook for this.

🔍 Root Causes

1. Stateful Data Modeled with Stateless Batch Jobs

Subscriptions have memory. Payments build on each other. But the analytics system pretended each batch was independent—forcing full history replays to reconstruct state.

2. OLAP Optimization Assumed Offline Preparation

Pinot’s speed came from precomputed aggregations. Remove that step for real-time data, and suddenly you’re doing complex windowed aggregations at query time—something the original engine couldn’t handle.

3. No Strategy for Incremental Schema Evolution

Metric definition changes were treated as “reindex everything from scratch” events. No concept of applying changes incrementally while preserving consistency.

🧠 The Solution Architecture

1. Event-Driven Pipeline with Flink State Management

Apache Flink now maintains a compressed “state” for each subscription—an ongoing ledger of its history. New events update that state incrementally. No more replaying billions of events just to add one payment.

The Cold Start Problem: New Flink deployments still needed initial state. Solution: A custom tool runs streaming transformation logic as a batch job in Apache Spark, which processes historical events in parallel and outputs verifiable flat files. This seeds Flink state efficiently and feeds a redundant offline pipeline for validation.

Result: Latency dropped from 24 hours to 15 minutes. The June payment is just appended to the ledger, not recalculated against six months of data.

2. Apache Pinot v2 Query Engine with Windowed Aggregation

Stripe needed to aggregate subscription data at query time without preaggregation—but maintain sub-second response times.

The breakthrough: Pinot’s open-source maintainers released a v2 engine with windowed aggregation queries. These segment data into time windows and perform aggregations (sum, average, etc.) across those ranges in real time.

This unlocked:

  • MRR calculations over arbitrary date ranges without offline prep

  • Complex data joins for currency conversion and gap filling

  • Custom query dimensions on live data

Stripe worked directly with Pinot maintainers to production-test this engine at scale—something never deployed in a user-facing system at Stripe’s volume before.

Result: Query latency under 300ms. Updates available to users within 15 minutes. Dashboard stays fast and responsive.

3. Dual-Mode Metric Definition Updates

When a customer changes an MRR formula, Stripe can’t just stop the world and reprocess. The solution is a carefully orchestrated workflow:

Step 1: Kick off batch job to recalculate all historical data with new definition
Step 2: Continue streaming new events using old definition in real time
Step 3: Buffer incoming events temporarily in Flink application memory
Step 4: Once historical recalc finishes, patch Flink state with updated history
Step 5: Replay buffered events on top of the new historical state
Step 6: Atomically switch dashboard to fully updated view
Step 7: Stop all processing with old definition

Critical constraint: The dashboard never shows inconsistent data. No gray screens. No “processing...” banners. Customers see a coherent view from history to present—even while definitions change and new data streams in.

🧰 The Cascade of Benefits

Before: Batch processing → 24hr lag → preaggregated queries → custom definitions break everything

After: Flink streaming → 15min latency → on-the-fly aggregation → definition changes without downtime

Unlocked features:

  • Real-time trial conversion tracking (spot trends within minutes)

  • Currency conversion and data gap filling in live queries

  • Flexible metric definitions that don’t require downtime

  • Historical accuracy maintained even as business changes

🤔 Lessons Learned

1. Stateful Data Demands Stateful Architecture

Treating subscriptions as stateless batch jobs was a fundamental mismatch. Streaming state management (Flink) aligns the architecture with the domain model.

2. OLAP Speed Requires Modern Query Engines

Legacy Pinot couldn’t do real-time windowed aggregation. Waiting for—and helping ship—the v2 engine was the only path forward. Sometimes you need to push your infrastructure forward, not just use what exists.

3. Schema Evolution Needs First-Class Support

Custom metric definitions weren’t an edge case—they were core to product value. The dual-mode update system should’ve been designed from day one, not bolted on later.

4. Real-Time Doesn’t Mean “Throw Out Batch”

The Spark-based historical processor and offline pipeline aren’t just for migration—they’re validation infrastructure. Streaming systems need batch counterparts for verification and recovery.

5. Consistency Beats Uptime in Analytics

Stripe chose to never show inconsistent data, even if it meant complex buffering and dual-processing. For financial analytics, a 5-minute delay is better than showing wrong numbers.

6. Partner with Open Source When You’re at the Edge

Pinot’s v2 engine existed but wasn’t battle-tested at Stripe’s scale. Working directly with maintainers to production-harden it benefited everyone: Stripe got real-time analytics, the community got a proven engine.

🏗️ What Stripe Built to Make This Work

  • Apache Flink state management for incremental subscription updates

  • Spark-based bootstrap tool for parallel historical event processing

  • Apache Pinot v2 production deployment with windowed aggregation

  • Dual-mode metric definition update system with buffering and patching

  • Redundant offline pipeline for validation and data export

  • Sub-300ms query latency on live, non-preaggregated data

🏁 Bottom Line

Stripe didn’t rebuild billing analytics because it was broken. They rebuilt it because customers were making decisions on stale data, and in 2024’s fast-moving markets, 24-hour-old data loses.

For engineering leaders and architects:

  • Align architecture with domain statefulness. Subscriptions have memory. So should your analytics.

  • Invest in query engine capabilities. Pinot v2’s windowed aggregation was the unlock. Know when to wait for infrastructure vs. build around it.

  • Design for schema evolution upfront. Custom definitions, changing business rules, metric adjustments—these aren’t edge cases.

  • Consistency > availability in financial systems. Never show wrong numbers, even if it means brief delays.

  • Keep batch as a validation layer. Streaming is primary, but batch processing validates correctness and enables recovery.

  • Work with open source at scale. Stripe and Pinot both won by collaborating on production-hardening v2.

Real-time billing analytics is about making accurate data available when decisions are being made. Stripe built a system where “when” is now.
